## 1. 方案摘要
本方案采用“前后端分离但仓内协同”的 V1 路线：以 Web 前端承载棋盘、讨论区、登录页与后台页面；以后端 API 承载用户、对局、模型配置、运行策略与审计能力；以独立规则适配层承载象棋合法性与终局判断；以 AI 策略 / 演绎层承载难度控制、候选步解释、角色化文本生成。

核心取舍是：
- 规则正确性优先，不让大模型直接决定合法性
- 演绎体验做成产品亮点，但不暴露原始思维链
- V1 不做人对人联机，不上复杂实时系统
- 架构上预留未来 Qt 客户端、人人对战、棋局恢复的扩展位
- 技术上优先选择 TypeScript 全栈，以降低 Web 与规则适配、后台管理和未来 API 复用成本

## 3. 技术上下文
- 语言 / 运行时：TypeScript；Node.js 22；浏览器端 TypeScript / React
- 核心依赖：React、Vite、Node.js Web API 框架（推荐 Fastify）、数据库 ORM（推荐 Prisma 或 Drizzle）、规则库 baseline（优先评估 xiangqi.js）
- 数据存储：SQLite 作为 V1 默认本地持久化存储；后续保留迁移到 MySQL / PostgreSQL 的空间
- 测试方式：单元测试 + 规则场景测试 + API 集成测试 + 前端主链路冒烟测试
- 目标运行环境：Linux 服务器部署；桌面浏览器与移动浏览器访问
- 项目类型：单仓全栈 Web 应用（前端 + 后端 + 共享类型 / 规则适配层）
- 性能 / 资源约束：优先保证规则响应与主链路稳定；AI 演绎内容控制在短文本；限制 AI 并发与单账号进行中对局数
- 范围 / 规模预估：V1 为单产品闭环，不追求多人联机和高并发平台化能力

## 4. 总体方案设计
### 主链路
1. 用户登录进入对局页
2. 前端展示棋盘、主题、操作区和讨论区
3. 用户选择棋子并提交落子
4. 后端规则适配层校验合法性并更新局面
5. 若用户落子非法，返回风格化提示模板输入，前端仅做展示
6. 若用户落子合法，后端生成 AI 合法候选步集合
7. AI 策略层根据当前难度和策略规则，在候选步中选出最终落子
8. AI 演绎层围绕本回合生成角色化展示内容
9. 后端返回：新局面、回合展示、事件摘要、剩余悔棋次数、对局状态
10. 前端按“逐段出现”的方式展示回合结果

### 关键流程 / 数据流
- 棋局数据流：UI 操作 → API → RuleAdapter → GameSession 更新 → API 返回新状态
- AI 决策流：GameSession 当前局面 → RuleAdapter 合法候选步 → Strategy 层筛选 → Narrative 层包装 → 返回最终落子与展示数据
- 后台配置流：管理员修改模型 / 运行策略 → 后端校验与持久化 → 审计记录 → 运行期读取生效
- 登录与偏好流：用户认证成功 → 拉取偏好与最近对局 → 前端初始化主题 / 设置 → 后续变更回写数据库

## 5. 模块划分
### 模块 1：Web 前端（用户端）
- 职责：登录、棋盘交互、讨论区展示、主题切换、对局操作、最近对局摘要展示
- 输入 / 输出：调用后端 API；接收对局状态、回合展示、用户偏好与历史数据
- 依赖：认证 API、对局 API、偏好 API
- 对应用户故事 / FR：用户故事 1 / 2 / 3 / 4；FR-001 ~ FR-019、FR-028、FR-029

### 模块 2：后台前端（管理端）
- 职责：用户管理、模型配置、运行策略配置、审计摘要查看
- 输入 / 输出：调用后台 API；展示配置状态与测试结果
- 依赖：管理员认证、后台管理 API
- 对应用户故事 / FR：用户故事 5；FR-002、FR-023 ~ FR-027

### 模块 3：认证与用户服务
- 职责：用户名密码登录、账号状态控制、用户偏好、最近对局摘要
- 输入 / 输出：接收认证请求、返回用户会话与偏好数据
- 依赖：数据库、审计模块
- 对应用户故事 / FR：用户故事 4 / 5；FR-001、FR-002、FR-005、FR-008、FR-009

### 模块 4：对局服务（GameSession）
- 职责：创建 / 恢复当前进行中对局、落子处理、悔棋、认输、终局状态管理
- 输入 / 输出：接收用户动作，输出更新后的棋局状态和事件流
- 依赖：RuleAdapter、Strategy / Narrative、数据库
- 对应用户故事 / FR：用户故事 1 / 2 / 3；FR-006、FR-009 ~ FR-017

### 模块 5：规则适配层（RuleAdapter）
- 职责：封装规则库，提供合法走法生成、落子校验、终局判定、悔棋支持
- 输入 / 输出：输入局面与动作，输出规则判断结果
- 依赖：成熟 JS 规则库 baseline（优先评估 xiangqi.js）
- 对应用户故事 / FR：用户故事 1 / 2 / 6；FR-010 ~ FR-014

### 模块 6：AI 策略层（Strategy）
- 职责：根据难度控制候选步选择，保证最终落子来自合法集合
- 输入 / 输出：输入当前局面、合法候选步、运行策略；输出最终落子与简短决策摘要
- 依赖：RuleAdapter、模型配置、运行策略
- 对应用户故事 / FR：用户故事 1 / 6；FR-006、FR-014、FR-023 ~ FR-026

### 模块 7：AI 演绎层（Narrative）
- 职责：生成每回合角色化展示、特殊事件模板、终局评价
- 输入 / 输出：输入局面上下文、主题、角色选择与最终落子；输出可公开展示的短文本结构
- 依赖：模型配置、主题模板、角色规则
- 对应用户故事 / FR：用户故事 1 / 3；FR-018 ~ FR-022

### 模块 8：后台配置与审计服务
- 职责：模型配置持久化、运行策略生效、审计记录、未配置状态管理
- 输入 / 输出：输入管理员操作；输出生效结果、测试结果、未配置提示状态和审计摘要
- 依赖：数据库、模型连通性测试器
- 对应用户故事 / FR：用户故事 5；FR-023 ~ FR-029

## 6. 目录结构 / 结构决策
### 推荐目录结构
- `src/frontend/`：用户端前端
- `src/admin/`：后台前端
- `src/server/`：后端 API 与服务入口
- `src/shared/`：共享类型、常量、主题定义、DTO
- `src/domain/game/`：GameSession、规则相关领域对象
- `src/domain/rules/`：RuleAdapter 与规则测试集
- `src/domain/ai/strategy/`：难度与候选步选择逻辑
- `src/domain/ai/narrative/`：角色演绎与事件模板
- `src/domain/user/`：用户、认证、偏好、对局摘要
- `src/domain/admin/`：后台配置、审计、策略管理
- `docs/`：需求、规则附录、后续方案文档
- `tests/`：规则测试、API 测试、主链路测试

### 结构决策说明
- 采用领域分层而不是把所有逻辑塞进页面层，目的是为未来 Qt 客户端和联机扩展留出 API 与共享领域层空间
- RuleAdapter 与 AI Strategy / Narrative 分离，防止规则正确性被演绎或模型调用污染
- 后台与用户前端分目录，避免管理能力与用户主链路混杂
- 共享类型与 DTO 单独收敛，方便前后端与未来客户端复用

## 7. 实施路径（MVP 路径）
1. 先做：项目基础框架、用户认证、规则适配层 baseline、基础对局主链路
2. 再做：AI 合法候选步选择、回合演绎展示、悔棋 / 认输 / 终局评价、手机端适配
3. 最后做：后台模型配置、运行策略、轻量审计、主题系统、最近对局摘要与偏好保存

### 最小可演示切片
- 单用户登录
- 新建一局人机对战
- 用户合法落子 → AI 合法应对 → 展示一轮角色化讨论
- 支持非法落子提示、认输、基础终局评价

## 8. 风险与取舍
### 主要风险
- 规则库 baseline 对长将 / 长捉 / 循环等细则支持不完整，需要后续附录与测试集兜底
- 若过度依赖大模型承担策略与演绎双职责，可能带来成本与稳定性问题
- 手机端讨论区若设计不当，会压制棋盘可用性
- 若后台模型配置缺失、异常或未完成初始化，前台对局与管理体验需要有明确的“未配置”反馈，否则易出现静默失败和定位困难

### 主要取舍
- 为什么采用当前方案：它在规则正确性、产品可玩性和后续扩展性之间平衡较好，适合 V1 快速形成闭环；同时把模型运行配置统一收敛到后台，有利于降低运维歧义
- 为什么不采用更复杂方案：不做人对人联机、不上复杂实时系统、不把完整存档和高级运营功能一起塞入 V1，避免项目爆炸；同时不采用 env-like + 后台双主配置源，避免配置优先级混乱

## 9. 复杂度与简化说明
### 本轮保留的必要复杂度
- 严格规则层与规则附录预留
- AI 策略层与演绎层分离
- 用户体系、后台配置和审计基础能力
- 多端可用与未来 Qt / 联机预留边界

### 本轮刻意不引入的复杂度
- WebSocket 实时联机体系
- 完整棋局恢复与云端棋谱管理
- 多角色权限平台
- 复杂 BI / 运营分析平台
- 读秒 / 用时制
