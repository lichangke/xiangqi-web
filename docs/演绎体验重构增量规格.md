## 1. 背景与目标
### 背景
当前项目已经完成了“演绎链路可工作”“统一时间线已形成”“decision / narrative 主链路已接通并完成 burn-in 收口”的一系列工作，D-2.8 也已正式收口。

但真实体验反馈表明：
当前演绎的主要问题，不再是“有没有接上模型”“链路是否稳定”或“还缺不缺 burn-in 样本”，而是**当前正式约束下产出的演绎产品形态本身，不是目标体验**。

目前正式约束仍明显偏向：
- 每回合固定可见结构
- 简评 / 发言 / 共识 / 落子的明确分段
- 2～4 个角色的短发言
- 偏克制、偏短促的回合表达
- 对“落子结论”的显式可见强调

这套结构虽然利于控长、控成本和保证结构整齐，但实际体验下来更像“结构化短演绎”，而不是“真正的剧情演出”。

### 目标
本轮目标不是继续微调 prompt，也不是继续验证旧结构，而是重新定义演绎的产品目标与可见层结构，使其从“结构化短演绎”升级为：

**以角色发言和群像互动为主体、能随棋局推进形成连续戏剧张力的话剧式演绎体验。**

本轮的核心收口点是：
1. 普通回合的主要可见层，到底应以什么结构呈现；
2. “简评 / 共识 / 落子”是否应删除、弱化、隐藏或改造成非主视图信息；
3. 角色个性、角色互动和整局连续性在第一版里至少要做到什么程度；
4. 新方向下哪些旧约束应被视为待重审，而不再继续当作默认前提；
5. 在演绎成为项目亮点的前提下，成本策略应如何正式向“效果优先”倾斜。

本轮**只解决增量规格定义**，不直接进入实现交接。

## 2. 用户场景 / 用户问题
### 目标用户
1. 希望在下棋过程中获得明显剧情体验、而不只是看一段结构化解说的用户；
2. 对角色群像、性格碰撞、阵营气质和戏剧推进感敏感的用户；
3. 希望“看 AI 演戏”和“下棋推进”两件事能真正融合，而不是彼此割裂的用户。

### 使用场景
1. 用户完成一次合法落子后，希望看到的不是简短点评，而是各角色围绕当前局势展开像样的对话；
2. 棋局进入关键转折、将军、换子、压迫或收束阶段时，用户希望明显感到戏剧张力提升，而不是仍然套同一模板；
3. 连续观看 3～5 个回合时，用户希望感到这是同一场戏在推进，而不是每回合重新开机；
4. 即使在手机端，用户也希望演绎更精彩，但仍然能顺畅阅读、不遮挡主链路。

### 当前问题
1. 当前回合可见结构把“简评 / 共识 / 落子”放在太核心的位置，角色发言反而像中间插段；
2. 当前角色发言偏短、偏弱、偏功能化，角色个性不够鲜明；
3. 当前演绎像“围绕棋步做结构化说明”，而不是“围绕局势展开戏剧互动”；
4. 当前连续性锚点虽然存在，但还不足以支撑用户感知到“整局是一场精彩的话剧”；
5. 旧约束中的字数、段数、固定四段结构和克制表达方向，正在把演绎稳定地导向一个用户明确不满意的形态；
6. 旧成本策略仍偏“有亮点但要克制”，而用户当前已明确：**演绎生成这条链路的模型成本上限可以设得很高。**

## 3. 用户故事（按优先级）

### 用户故事 1（P1）
- 故事描述：作为普通用户，我希望每个有效回合的可见主体是角色之间有来有回的发言与互动，而不是先看一套简评、共识和落子结论。
- 优先级原因：这是当前体验落差最大的核心点，直接决定演绎产品是否成立。
- 独立价值：即使只先重构普通回合展示结构，也能明显改变产品感受。
- 独立验证方式：连续查看多个回合时，用户主观感受应首先是“在看人物对戏”，而不是“在看分段解说”。

### 用户故事 2（P1）
- 故事描述：作为普通用户，我希望不同棋子角色真的有稳定、可辨认的性格与说话方式，而不是都在同一种模板腔里轮流发言。
- 优先级原因：没有鲜明角色，就不会有真正的群像戏感。
- 独立价值：即使剧情主线还不复杂，鲜明角色本身也能提升记忆点与可玩性。
- 独立验证方式：在不看角色标签的情况下，用户仍能从语言风格中大致分辨至少 3 类角色。

### 用户故事 3（P1）
- 故事描述：作为普通用户，我希望随着棋局推进，演绎能明显体现局势变化、关系升温、压力累积或节奏反转，而不是每回合都像独立小段子。
- 优先级原因：没有连续推进感，就无法形成“整场戏”的体验。
- 独立价值：只要连续性成立，用户就更愿意继续看下去，而不是看一两回合就跳过。
- 独立验证方式：连续 3～5 回合的演绎读下来，用户能明确感知到一条持续推进的局势主线。

### 用户故事 4（P2）
- 故事描述：作为普通用户，我希望最终落子信息仍然可理解，但不应强行以“落子结论段”的方式压过剧情主体。
- 优先级原因：棋局信息仍要清楚，但它不应继续主导演绎产品形态。
- 独立价值：把事实与戏感平衡好，才能既好看又不失真。
- 独立验证方式：用户能理解 AI 做了什么，但不会主观觉得页面重点仍在“公告结果”。

### 用户故事 5（P2）
- 故事描述：作为用户，我希望即使普通回合也能获得明显高质量的剧情演绎，而不是因为节省成本被压成过短、过弱、过保守的输出。
- 优先级原因：如果成本顾虑继续主导普通回合，体验会再次被收缩回旧形态。
- 独立价值：哪怕特殊事件还没完全打磨到位，普通回合一旦变得好看，产品辨识度就会明显提升。
- 独立验证方式：普通回合也能持续产出具有角色感和戏剧推进感的内容，而不是只有高光回合才“偶尔精彩”。

### 用户故事 6（P3）
- 故事描述：作为未来维护者，我希望新的演绎结构仍保持“系统定事实、AI 只负责表达”的硬边界，以便后续可持续演进。
- 优先级原因：演绎升级不能以牺牲规则可信性和事实稳定性为代价。
- 独立价值：能避免为了追求戏剧效果而把系统重新做脆。
- 独立验证方式：新规格中仍能清楚看到事实层、角色层、连续性层与可见展示层的边界分离。

## 4. 验收场景
### 场景 1：普通回合的主可见层改为“角色互动优先”
- Given：用户完成一次合法落子并进入 AI 回合
- When：系统返回该回合的演绎结果
- Then：用户首先看到的应是围绕当前局势展开的角色互动，而不是固定四段式的点评与结论展示

### 场景 2：角色个性在连续回合中可被辨认
- Given：同一局棋已经推进多个回合
- When：用户连续阅读多个回合演绎内容
- Then：至少核心角色的说话风格、判断方式和出场气质应具有可辨认差异

### 场景 3：棋局推进能带出连续戏感
- Given：棋局从试探进入压迫、换子、将军或收束阶段
- When：系统展示后续数回合演绎内容
- Then：用户应能感到节奏、语气、局势关系与戏剧张力在推进，而不是每回合彼此断裂

### 场景 4：落子事实继续存在但退居辅助层
- Given：AI 已完成最终合法落子
- When：用户查看该回合演绎
- Then：用户仍能理解这一步发生了什么，但“最终落子说明”不应再作为压过剧情主体的主展示块

### 场景 5：手机端仍可阅读
- Given：用户在手机端查看演绎
- When：普通回合演绎文本变得更丰富
- Then：页面仍应保持主链路可读、可滚动、不过度遮挡棋盘，且不会因为文案膨胀而直接失控

### 场景 6：普通回合也能持续获得高质量演绎
- Given：当前回合不属于“终局 / 将军 / 反杀”之类的特殊高光事件
- When：系统生成该回合演绎
- Then：输出仍应具有明显角色感、剧情推进感和可读性，而不应因为成本约束被压成贫弱模板

## 5. 本轮优先收口结论（接近可进 Plan 的推荐版本）

### 5.1 普通回合的新可见结构
当前推荐把普通回合改成 **“对戏主体 + 收束句 + 弱事实尾注”** 三层结构，而不是旧的四段式结构。

#### 主层：对戏主体
- 这是普通回合的核心可见层；
- 由 3 段左右连续角色发言构成，默认表现为“角色之间有来有回的戏剧互动”；
- 文本重点放在：局势判断、阵营态度、角色碰撞、压力变化、下一步意图，而不是复述棋步坐标；
- 页面不再显式显示“发言”这个段标签，直接把内容当正文主体展示。

#### 次层：单句收束
- 用 1 句简短收束当前回合的戏剧结论；
- 它承接旧“共识”功能，但不再以“共识 / 拍板”标签高亮出现；
- 这句的作用是帮助回合落地和稳住节奏，而不是像公告板一样宣布结果。

#### 辅助层：弱事实尾注
- 用极弱可见方式保留“AI 最终落子 / 本回合关键事实”；
- 推荐表现为：尾注、次级小字、时间线元信息或可折叠说明；
- 默认不再作为正文主段单独出现。

### 5.2 “简评 / 共识 / 落子”的处理建议
- **简评：删除主视图地位。**
  - 它最容易把用户拉回“解说视角”；
  - 若需要保留，可仅作为内部中间字段或摘要生成材料，不作为用户主视图块。
- **共识：弱化并改造成“收束句”。**
  - 保留其节奏收束价值；
  - 但不再维持“共识”标签和显式板块感。
- **落子：保留事实，不保留主段。**
  - AI 最终做了什么仍然必须可理解；
  - 但表现方式改为弱事实尾注或嵌入收束句，不再占据正文主体。
- **发言：从中间段升级为主体。**
  - 真正让用户“看到戏”的部分，应当就是这里。

### 5.3 第一版每回合的推荐组织方式
- 默认出场角色数：**3 个**；
- 关键回合允许升到 **4 个**；
- 仅在平淡回合、移动端极窄布局或降级情况下才退到 **2 个**；
- 每回合建议存在 1 个主导推进角色、1 个回应或制衡角色、1 个负责把局势拉回整局主线的角色；
- 帅 / 将不必每回合都出场，只在需要定调、收束或中枢风险升高时出场，避免回回抢戏。

### 5.4 对戏主体的最终表现形式
当前推荐：**以对白为主，允许极少量场记型句子，但场记只能服务于节奏与动作感，不能抢正文主体。**

具体口径：
- 普通回合正文默认以角色对白推进；
- 允许插入极少量的场记型短句，用于交代动作、空气变化、局面压迫感或转场；
- 场记句只能是短句、轻量、低频；
- 不允许把正文重新写回“旁白解说文”；
- 一回合中若已存在足够强的角色对白，优先不要再额外补场记。

这意味着：
- 不是走“纯对白到完全无锚点”的极端；
- 也不是回到“解说句 + 发言拼接”的旧路。

## 6. 角色个性强度的推荐口径
### 6.1 第一版不追求“夸张戏精化”
当前推荐的角色个性强度应为：**明显可辨认，但不过火，不做夸张人设表演。**

换句话说，第一版要做到的是：
- 用户能明显分辨“谁在说话”；
- 不同棋子类型与不同阵营气质能稳定拉开；
- 角色说话方式和判断关注点持续一致；
- 但不把角色写成脱离棋局的夸张漫画人物。

### 6.2 第一版建议优先拉开的四个维度
1. **关注点差异**
   - 车更看压线、通道、正面推进；
   - 马更看变线、侧击、空隙；
   - 炮更看借势、支点、时机；
   - 士 / 象更看中枢、结构、边界；
   - 兵 / 卒更提供前线体感；
   - 帅 / 将负责少量定调与决断。
2. **句式差异**
   - 有的角色短促、直接；
   - 有的角色更审慎、慢一点；
   - 有的角色更像判断者；
   - 有的角色更像一线观察者。
3. **阵营气质差异**
   - 丹陛城更偏稳局、守中见锋；
   - 玄旌府更偏压迫、夺势、收网；
   - 同一棋子类型在不同阵营下，应带有同底色但不同战术气味。
4. **出场职责差异**
   - 不是所有角色都负责“分析”；
   - 有的角色负责推进火力，有的负责拉回风险，有的负责补现场感，有的负责落最终调子。

### 6.3 第一版不建议拉开的维度
- 不做过强口头禅化；
- 不做夸张喜剧化区分；
- 不把角色个性建立在大量古风辞藻或设定堆砌上；
- 不追求每个角色都有厚重个人史。

## 7. 连续戏感机制的推荐口径
### 7.1 当前判断
旧的 `storyThreadSummary` 思路有价值，但**单靠一个偏摘要化的文本对象，不足以支撑“话剧式推进感”**。

原因在于：
- 它能概括最近几回合在讲什么；
- 但不够直接回答“这回谁该接着说”“当前紧张关系挂在哪根线上”“下一回合最该延续哪种戏剧压力”。

### 7.2 推荐升级方向
当前建议把旧的 `storyThreadSummary` 升级理解为：
**`StoryState`（戏剧状态对象） + 面向模型的短摘要表达**。

也就是说：
- 底层维护的是更结构化的状态；
- 给模型时可以再转成短摘要或混合输入；
- 但系统内部不再只靠一段摘要文字承载全部连续性。

### 7.3 第一版推荐最少槽位
在原有 `currentPhase / mainConflict / pressureSide / recentFocus / carryForward` 之外，新增至少两个槽位：
- `activeTension`：当前最值得继续拉扯的紧张关系，例如“中路压迫未卸”“边线试探转正面逼位”；
- `speakerMomentum`：下一回合最该持续发声的角色类型或阵营方向，例如“黑方车炮继续主导话语权”“红方士象需要接过防守判断”。

因此第一版推荐结构为：
```json
{
  "currentPhase": "string",
  "mainConflict": "string",
  "pressureSide": "RED | BLACK | BALANCED",
  "recentFocus": "string",
  "carryForward": "string",
  "activeTension": "string",
  "speakerMomentum": "string"
}
```

### 7.4 这套机制要解决什么
- 解决“连续几回合像同一场戏”的问题；
- 解决“角色虽然有发言，但每回合像随机出镜”的问题；
- 解决“局势明明在升温，文案却仍像平铺说明”的问题。

### 7.5 当前建议
- `StoryState` 仍优先由系统规则化更新，不新增第三个轻量 AI 环节；
- 旧 `storyThreadSummary` 可退化为 `StoryState` 的面向模型简述层，而不再是唯一连续性载体。

## 8. 事实显隐策略与移动端呈现口径
### 8.1 原则
事实必须保留，但**事实的呈现层级必须后退**。

### 8.2 推荐分层
- **第一层：戏剧正文**
  - 用户首先看到角色互动和局势演出。
- **第二层：单句收束**
  - 用户快速理解本回合态势落点。
- **第三层：事实尾注 / 元信息**
  - 用户需要时仍能看到最终落子、关键变化、特殊标记。

### 8.3 不建议的做法
- 不再把“AI 落子：XXX，理由：YYY”做成正文主句；
- 不再把“点评 + 结论”放在角色发言前后形成夹心结构；
- 不再为了保证“信息清楚”而牺牲主要阅读体验。

### 8.4 弱事实尾注在移动端的推荐表现
当前推荐采用：**常驻单行摘要 + 可展开详情**。

具体口径：
- 移动端默认常驻一行弱事实摘要，例如“黑方落子：炮八平五｜中路继续施压”；
- 这一行默认以次级小字呈现，不抢正文主体；
- 若用户需要更多细节，可点击展开完整事实说明；
- 不推荐把全部事实完全藏进纯点击展开，否则容易让用户失去最基本的局势把握；
- 也不推荐把事实完整摊平，否则又会压回旧结构。

### 8.5 普通回合在手机端的长度与折叠策略
当前推荐：**生成可以更长，但显示默认分层，不一次性全摊。**

建议口径：
- 普通回合的模型生成长度不再沿用旧的 120～180 字硬约束；
- 手机端默认首屏可见正文建议控制在约 220～320 字级别；
- 若本回合为关键回合或文本本身更精彩，可允许完整正文达到更高上限，但首屏应保留折叠 / 展开能力；
- 关键回合与特殊事件可明显高于普通回合的显示长度；
- 真正要控制的不是“模型绝对输出字数”，而是“默认首屏阅读负担”。

## 9. 特殊事件与新结构的结合方式
### 9.1 当前推荐
特殊事件不建议完全照搬普通回合“对戏主体 + 收束句 + 弱事实尾注”的节奏，
而应采用：**“裁断锚点 + 角色余波 + 弱事实尾注”** 的事件结构。

### 9.2 含义
- **裁断锚点**：先用极短的一句把事件性质钉住，例如“此议不准”“前线回拨”“将势已成”“棋局已终”；
- **角色余波**：随后让相关角色对这次事件作出反应、消化和延续，把事件重新接回整局戏剧主线；
- **弱事实尾注**：保留规则原因、状态变化、关键事实说明。

### 9.3 为什么不用纯“角色即时反应”
因为特殊事件本身常带有强事实性和强状态变化：
- 非法步需要明确“未通过”；
- 悔棋需要明确“已回拨”；
- 将军需要明确“威胁成立”；
- 终局需要明确“已结束”。

如果完全让角色反应吞掉事件锚点，用户会更难把握实际状态。

### 9.4 为什么也不能退回“纯裁断说明”
因为那样又会把特殊事件写回公告文，不符合本轮要建立的角色驱动方向。

因此当前推荐的平衡点是：
- 先短促定性；
- 再用角色余波把戏延续下去；
- 最后用弱事实尾注补足理解。

## 10. 演绎生成的成本与调用策略口径
### 10.1 新的总原则
当前已确认：**演绎生成这条链路的模型成本上限可以设得很高。**

这意味着：
- 演绎生成的默认策略，不应再以“尽量省成本”作为第一原则；
- 普通回合也可以为了效果投入显著更高的模型成本；
- 只要不破坏主链路稳定性，成本应优先服务于演绎质量，而不是反过来用成本约束把体验压回去。

### 10.2 具体口径
- 不对同一局演绎设置总预算上限；
- 不对连续多回合演绎设置严格总量上限；
- 普通回合允许使用高质量模型，而不默认只给“轻量档”；
- 高光事件与特殊事件仍可进一步提升表现档位，但普通回合不应因节省成本而明显降级成弱模板；
- 若确实需要收缩，优先收缩显示层长度与非关键补充，而不是优先削弱模型能力本身。

### 10.3 仍需保留的边界
- 成本可以高，不等于可以无限等待；
- 成本可以高，不等于可以牺牲主链路响应和用户操作闭环；
- 成本可以高，不等于可以取消 schema 约束、事实约束与 fallback 机制；
- 成本优先级提升，不代表普通回合必须无上限铺陈长文。

### 10.4 当前建议
后续在 Plan 阶段，应把旧的 `docs/Task Bundle C 成本与调用策略.md` 作为**待重审文档**：
- 保留“演绎是亮点能力”的基本判断；
- 但将其从“预算优先花在高光时刻”升级为“普通回合也可高投入，高光事件再进一步加强”。

## 11. 边界情况 / 异常场景
- 当模型生成内容过于松散、偏离局势或角色失真时：系统必须有约束与降级策略，不能任由输出自由失控；
- 当演绎需要变得更丰富时：不得把它直接放大成无限长文，仍需保证节奏、阅读负担与移动端可用性；
- 当角色发言更有戏感时：不得牺牲事实正确性，不得让 AI 自造不存在的棋局结果；
- 当本轮重构普通回合结构时：不得顺手把“对局 AI 智能提升”“外部访问验证”“公网巡检”重新混进本轮边界；
- 当旧文档中的约束与本轮方向冲突时：应明确标记为“待重审 / 待 supersede”，而不是继续隐性沿用。

## 12. 功能需求（FR）
- FR-001：普通回合的用户可见层必须从“结构化结论优先”切换为“角色互动优先”。
- FR-002：普通回合的推荐主结构应为“对戏主体 + 单句收束 + 弱事实尾注”，不再强制保留“简评 / 发言 / 共识 / 落子”的固定四段可见结构。
- FR-003：普通回合正文应以角色对白为主，允许极少量场记型短句，但不得回退为旁白解说主导。
- FR-004：角色发言必须成为普通回合的主体内容，而不是夹在点评与结论之间的中间段。
- FR-005：`简评` 不再保留主视图地位；如需保留，仅可作为内部中间字段或摘要生成材料。
- FR-006：旧 `共识` 应改造成不带强标签的单句收束层，而不是独立公告式区块。
- FR-007：`落子` 相关事实必须保留，但默认以下沉为尾注、元信息或嵌入式表达为主，不再占据正文主体。
- FR-008：系统必须让角色个性在多回合内保持稳定且可辨认，包括关注点、句式、立场、判断方式与表达习惯。
- FR-009：第一版普通回合默认 3 个角色出场，关键回合可升到 4 个，平淡回合或受限场景可降到 2 个。
- FR-010：普通回合演绎必须显著锚定当前局势、当前回合变化与上一阶段剧情延续，避免泛泛空话。
- FR-011：系统必须支持“连续数回合仍像同一场戏”的演绎连续性，而不是每回合重新开机。
- FR-012：连续性机制不得再只依赖旧的摘要式 `storyThreadSummary`；系统必须维护更结构化的 `StoryState`，至少包含 `currentPhase / mainConflict / pressureSide / recentFocus / carryForward / activeTension / speakerMomentum`。
- FR-013：特殊事件应采用“裁断锚点 + 角色余波 + 弱事实尾注”的推荐结构，而不是纯公告说明或纯角色反应。
- FR-014：事实判定权仍全部归系统所有；AI 只负责表达，不拥有合法性、终局、事件成立与最终业务状态的裁定权。
- FR-015：本轮需重新定义普通回合与特殊事件的输出契约，以支撑更丰富的角色互动和更弱显式结论层。
- FR-016：手机端展示必须在演绎更丰富的前提下继续保持主链路可读，不得因内容升级直接破坏使用体验。
- FR-017：移动端弱事实层应默认采用“常驻单行摘要 + 可展开详情”的表现方式。
- FR-018：普通回合的模型生成不再受旧的 120～180 字硬约束；显示层应控制首屏阅读负担，而不是简单压缩生成能力。
- FR-019：演绎生成链路的模型成本上限可以设得很高；普通回合不应因节省成本而被默认压回低质量输出。
- FR-020：即使成本上限提高，系统仍必须保留超时边界、schema 校验、事实约束与 fallback 机制。
- FR-021：当模型输出失败、漂移或不符合新契约时，系统必须可降级，但降级方向不应再默认回到旧的固定四段主展示结构。

## 13. 非功能需求（NFR，可选）
- NFR-001：演绎质量的提升不得以事实漂移、规则失真或业务状态混乱为代价。
- NFR-002：新的可见结构应显著降低模板腔和公告腔，优先追求角色辨识度、戏剧推进感与阅读兴趣。
- NFR-003：连续回合之间的语言风格、角色气质与剧情主线必须保持一致性，避免明显断裂。
- NFR-004：即使内容更丰富，也必须保持移动端可读性、可滚动性与成本可控性；这里的“可控”优先指首屏阅读负担可控，而非模型成本必须压低。
- NFR-005：架构上仍需保持事实层、角色层、连续性层和展示层的边界分离，以便后续安全演进。
- NFR-006：在演绎生成链路上，效果优先级应高于节省模型成本；若两者冲突，默认优先保留体验效果。

## 14. 关键实体 / 核心对象（必要时）
- 回合演绎项（NarrativeTurn）：承载某回合面向用户展示的最终演绎结果，不再默认等同于旧四段结构。
- 角色发言单元（RoleLine）：表示某一角色在当前回合中的一条发言、态度或动作表达。
- 戏剧状态对象（StoryState）：用于维持跨回合连续性的结构化状态对象，是新连续性机制的核心。
- 角色卡（RoleCard）：定义角色的立场、气质、话语风格、行动倾向与禁区。
- 可见性策略（NarrativeVisibilityPolicy）：定义哪些事实以主内容展示、哪些事实退居辅助层、哪些事实嵌入剧情正文。
