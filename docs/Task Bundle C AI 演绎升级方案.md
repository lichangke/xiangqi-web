# Task Bundle C AI 演绎升级方案

## 文档定位
- 文档类型：增量方案草案
- 适用范围：Task Bundle C（演绎展示与移动端体验）后续修正与升级
- 当前状态：草案
- 最后更新时间：2026-03-27

## 1. 背景
当前 Task Bundle C 已完成真实代码实现，并通过专项测试、回归测试、构建验证与最小运行态验证，说明当前实现具备真实技术完成度。

但在真实手机人工走查中，已暴露出两个核心问题：
1. 非法步、悔棋等特殊事件虽然已有提示，但没有真正并入演绎区
2. 当前演绎整体偏呆板，产品感不足

这说明：
当前 C 的问题已经不只是“模板补得不够细”，而是演绎机制本身的表达能力不足。

## 2. 本次升级判断
当前建议将 Task Bundle C 的后续修正，从：
- “继续强化模板演绎”

升级为：
- **“AI 驱动的结构化剧情演绎系统 V1”**

本升级的核心前提是：
- 演绎是本项目的最大亮点之一
- 若后续还会引入更多剧情、角色与事件，仅靠模板会越来越难维护，且很难避免呆板化
- 因此应在当前阶段尽早把演绎能力从“模板收口”升级为“受控 AI 文案生成”

## 3. 当前已形成的方向共识
### 3.1 先固定一条剧情主线
第一版不做多世界观切换，而是先定一条固定剧情主线，用于承载整局对弈的叙事语境。

该剧情主线至少要明确：
- 整局像什么故事
- 双方是什么阵营或关系
- 整局叙事基调是什么
- 胜负在故事里意味着什么

### 3.2 先建立棋子角色卡
第一版不以“每一枚棋子的长期个体履历”为目标，而是先按棋子类型建立稳定角色设定。

建议覆盖：
- 帅 / 将
- 车
- 马
- 炮
- 相 / 象
- 仕 / 士
- 兵 / 卒

每类角色卡至少包含：
- 身份定位
- 性格特征
- 说话风格
- 常见行动风格
- 在剧情中的职责
- 不该说什么 / 不该表现成什么

### 3.3 棋步不只表示棋盘变化，还要表示剧情推进
第一版不建议让 AI 直接从原始棋步自由发挥，而应先由系统做一层“叙事语义解释”。

也就是说：
- 系统先判断这一步在叙事上更像什么
- 再把“棋步事实 + 叙事语义标签 + 世界观设定 + 角色卡”交给 AI 写文案

可先考虑的叙事语义标签包括：
- 试探
- 压迫
- 抢位
- 换子
- 解围
- 护驾
- 佯动
- 追击
- 收官

### 3.4 AI 只拥有文案权
本次升级的硬边界建议明确为：

> AI 只负责“怎么说”，不负责“发生了什么”。

也就是说：
- 系统负责：规则判定、事件判定、局势状态、时间线结构、叙事输入标签
- AI 负责：文案、语气、角色表达、剧情推进感与高光时刻表现

### 3.5 为了演绎效果，可以提升 AI 预算
当前建议接受：
- 演绎是亮点能力，因此可以提高 AI 成本预算

但建议是“结构化地贵”，而不是无节制地烧：
- 普通回合：允许 AI 参与，但输出应短、稳、结构化
- 高光回合 / 特殊事件：允许更高预算与更强表现

## 4. 第一版系统目标
本次 AI 演绎升级的第一目标不是“让 AI 自由讲故事”，而是：

**在不破坏对局主链路和规则可信性的前提下，让普通回合与特殊事件都能进入统一时间线，并由 AI 生成更有角色感、剧情感和产品感的文案。**

## 5. 第一版推荐架构
建议拆成三层：

### 5.1 事实层（非 AI）
由系统产出结构化事实，至少包括：
- 回合号
- 用户走子 / AI 走子
- 是否吃子
- 是否将军
- 是否终局
- 是否非法步
- 是否悔棋
- 当前主题
- 当前角色设定版本
- 当前叙事语义标签

这一层必须保持可控、可测、非 AI。

### 5.2 演绎生成层（AI）
把结构化事实 + 世界观 + 角色卡喂给 AI，让其返回结构化结果，而不是自由散文。

建议输出结构至少包括：
- title
- summary
- segments（用于逐段展示）
- tone
- highlightLevel
- optional event-specific fields

### 5.3 展示层（前端）
前端负责：
- 普通回合与特殊事件统一时间线展示
- 逐段出现
- 主题视觉切换
- 移动端布局
- fallback 展示策略

## 6. 第一版范围建议
### 明确包含
1. 一份固定剧情圣经（V1）
2. 一套棋子角色卡（V1）
3. 一层“棋步事实 → 叙事语义标签”的结构化输入
4. AI 生成普通回合演绎文案
5. AI 生成特殊事件演绎文案
6. 普通回合与特殊事件进入统一时间线
7. AI 调用失败时的 fallback

### 明确不包含
1. 多剧情世界切换
2. 后台剧情编辑器
3. 用户自定义角色系统
4. 让 AI 参与规则判定
5. 复杂长篇分支剧情系统
6. 后台可配置文案平台

## 7. 成本策略建议
### 7.1 普通回合
- 使用较轻但稳定的 AI 输出
- 输出短而结构化
- 不追求每回合都“大段高光”

### 7.2 特殊事件 / 高光回合
建议允许更高成本，包括但不限于：
- 非法步
- 悔棋
- 将军
- 终局
- 明显反杀或局势突变

### 7.3 长期建议
第一版实现时，架构应兼容“普通回合轻模型，高光事件强模型”的分层策略。

## 8. 对当前 Bundle C 的影响
这次升级意味着：
- 当前 C 的修正不再只是“改模板”
- 而是正式改为“AI 驱动的结构化演绎系统第一版”

因此后续建议：
- 不再把当前问题理解为“文案润色”
- 而是把它视为 Bundle C 的策略升级

## 9. 建议先落的三个设计产物
### 9.1 剧情圣经
建议后续单独起草，至少包括：
- 世界观
- 阵营关系
- 整体语气
- 各棋子角色设定
- 禁止出现的表达边界

### 9.2 演绎输入输出契约
建议后续单独起草，至少包括：
- 系统给 AI 的结构化输入字段
- AI 返回的结构化输出字段
- 普通回合 / 特殊事件的统一时间线模型
- fallback 触发条件与输出约束

### 9.3 成本与调用策略
建议后续单独起草，至少包括：
- 普通回合如何调
- 特殊事件如何调
- 超时 / 失败如何降级
- 是否缓存

## 10. 当前待确认事项
以下内容当前尚未正式确认，后续需要继续讨论：
1. V1 的剧情主线具体设定是什么
2. 各棋子的角色卡具体写法是什么
3. 叙事语义标签第一版先保留哪些
4. 普通回合与特殊事件是否使用同一模型档位
5. AI 输出的 JSON 契约字段具体长什么样
6. fallback 的文案策略与触发条件如何定义

## 11. 当前建议
当前建议不要立即进入实现，而是按以下顺序推进：
1. 先确认本升级方向本身
2. 再起草剧情圣经
3. 再起草演绎输入输出契约
4. 再起草成本与调用策略
5. 最后再据此更新 Bundle C 的执行协议与实现方案

## 12. 当前结论
本草案的核心结论是：

> 当前建议正式接受“为提升演绎效果而引入 AI”的方向，并将 Task Bundle C 的后续修正从“模板补强”升级为“AI 驱动的结构化剧情演绎系统 V1”。

但本文件当前仍为草案，待后续进一步确认后，再决定是否回写状态卡与执行协议。
