## 1. 方案摘要
本轮方案目标，不是继续在旧的四段式演绎上做局部润色，而是把当前演绎链路整体升级为：

- **普通回合**：以“对戏主体 + 单句收束 + 弱事实尾注”为核心可见结构
- **特殊事件**：以“裁断锚点 + 角色余波 + 弱事实尾注”为事件结构
- **连续性机制**：从旧 `storyThreadSummary` 升级为结构化 `StoryState`
- **显示策略**：移动端默认分层显示，不再靠压缩生成能力维持可读性
- **成本策略**：演绎生成链路默认转为“效果优先”，普通回合也允许高投入模型生成

核心取舍是：
1. **不动事实判定边界**：系统继续负责合法性、终局、事件成立与最终状态；
2. **重做演绎可见层与连续性层**：把用户真正看到的“戏”重构出来；
3. **旧文档不硬删，先作为待 supersede / 待重审输入处理**；
4. **优先走最短 MVP 路径**：先让普通回合结构与连续性机制成立，再收特殊事件与移动端细节，最后再固化成本 / 参数策略。

## 4. 总体方案设计
### 主链路
1. 用户完成合法落子
2. 系统完成事实计算与 AI 最终合法落子
3. 系统生成普通回合或特殊事件的结构化演绎输入
4. 系统更新 / 读取 `StoryState`
5. 演绎模型基于事实、角色卡、主题与 `StoryState` 生成结构化输出
6. 服务端校验 schema 并统一转成前端时间线项
7. 前端按新结构分层展示：正文 / 收束 / 弱事实层

### 关键流程 / 数据流
#### 普通回合
- GameService / decision 层给出：
  - 用户落子事实
  - AI 落子事实
  - 语义标签 / situation shift / turn arc 等中间结果
- Narrative 组装层：
  - 选择本回合出场角色
  - 读取上一版 `StoryState`
  - 生成普通回合 NarrativeRequest
- Narrative 生成层：
  - 返回对戏主体 / 收束 / 弱事实说明所需结构化字段
- 展示映射层：
  - 转换为统一时间线 item

#### 特殊事件
- 系统先产出事件事实与裁断锚点
- Narrative 组装层给出相关角色和 `StoryState`
- 模型生成角色余波内容
- 前端展示为“裁断锚点 + 角色余波 + 弱事实尾注”

### 当前推荐迁移方式
- 先在当前既有 `narrative-service.ts` / `narrative-fallback.ts` / `presentation.ts` 上引入 V2 分支；
- 等 V2 主链路成立后，再视需要把逻辑进一步拆成独立文件；
- 不建议一上来为了“结构好看”而先做大规模文件搬家，否则会拉长到达首个可见成果的时间。

### 模块 2：StoryState 管理层

  - 后续再拆出 `story-state.ts`

### 模块 3：Role Casting & Dialogue Planning
- 职责：决定本回合谁出场、谁主导、谁回应、谁负责收束
- 输入 / 输出：active roles, speaker order, role hints
- 依赖：角色卡、事件类型、局势标签、StoryState
- 对应用户故事 / FR：US1 / US2 / FR-003 / FR-007 / FR-008
- 当前建议承载点：
  - 首轮与 narrative service 同层实现；
  - 后续再拆成 `role-casting.ts`

### 模块 4：Narrative Provider Orchestration
- 职责：统一调用模型、做 schema 校验、做 fallback 和成本档位策略
- 输入 / 输出：NarrativeRequest → NarrativeResponse / fallback response
- 依赖：provider config, schema validators, timeout policy
- 对应用户故事 / FR：US5 / US6 / FR-018 ~ FR-021

  - 继续以 `apps/server/src/domain/ai/narrative/narrative-service.ts` 为主入口；
  - `narrative-fallback.ts` 承担 fallback-v2 的第一版实现

### 模块 5：Presentation Mapping
- 职责：把服务端返回映射为前端可直接消费的展示模型
- 输入 / 输出：timeline item（turn / event）
- 依赖：shared contract, web presentation model
- 对应用户故事 / FR：US1 / US4 / FR-001 ~ FR-007 / FR-017
- 当前建议承载点：
  - 首轮仍以 `apps/web/src/presentation.ts` 为主；
  - 等结构稳定后再拆组件层与映射层

### 模块 6：Web UI Layer Refresh
- 职责：把前端演绎卡片与时间线改成新分层结构
- 输入 / 输出：正文层、收束层、弱事实层、折叠行为
- 依赖：Presentation Mapping
- 对应用户故事 / FR：US1 / US4 / FR-016 / FR-017

## 6. 目录结构 / 结构决策
### 第一阶段推荐目录策略
第一阶段不强制先拆新文件，而是采用：
- **shared：先扩 `packages/shared/src/index.ts`**
- **server：先在现有 `narrative-service.ts` / `narrative-fallback.ts` 中引入 V2 逻辑**
- **web：先在现有 `presentation.ts` / `App.tsx` 中完成新展示层 MVP**

### 第二阶段推荐目录结构（当 V2 结构稳定后）
- apps/server/src/domain/ai/narrative/
  - contract-v2.ts
  - story-state.ts
  - role-casting.ts
  - provider-orchestrator.ts
  - fallback-v2.ts
- packages/shared/src/
  - narrative-contract.ts
  - narrative-presentation.ts
- apps/web/src/
  - presentation/
    - narrative-timeline.ts
    - narrative-card.tsx
    - narrative-fact-tail.tsx
    - narrative-event-card.tsx

### 结构决策说明
1. **不把新逻辑散落在旧模板代码里硬补丁**，但也不在第一步就为结构洁癖付出过大重构成本；
2. `StoryState` 不直接寄生在旧 `storyThreadSummary` 字符串上，而是先拥有结构化对象，再根据需要生成简述；
3. 前端展示改成组件分层，避免旧四段式 DOM 结构继续把新体验锁死；
4. shared package 承担契约与展示模型的公共类型，避免 server / web 各写一套。

## 7. StoryState 更新规则（推荐版本）
### 7.1 总原则
第一版 `StoryState` 采用**系统规则化更新优先**，不引入第三个 AI 更新器。

### 7.2 输入来源
每回合更新时只依赖：
- 当前回合事实结果
- decision 层已有中间结果：`userMoveTag / aiMoveTag / situationShift / turnArc / pressureSide / highlightReason`
- 上一版 StoryState
- 当前是否吃子 / 将军 / 终局 / 特殊事件

### 7.3 槽位更新建议
- `currentPhase`：优先由 `turnArc` + 事件状态驱动；若发生明显反转或终局，允许直接跃迁
- `mainConflict`：以当前回合 `situationShift` 为主，结合上一版冲突主线做短句更新
- `pressureSide`：优先使用 decision 层已有压力判断；若无，则由事件状态与局势标签兜底
- `recentFocus`：由最近 1～2 回合最显著的冲突焦点更新，不追求长历史回放
- `carryForward`：永远面向“下一回合最该延续什么”，禁止写成复盘总结
- `activeTension`：若当前有将军、关键压迫、换子后的再布势、未卸压等强冲突，则优先体现
- `speakerMomentum`：根据当前更该主导发言的阵营 / 棋子类型决定，为下一回合 Role Casting 提供方向

### 7.4 更新优先级
建议优先级：
1. 特殊事件 / 终局 / 将军
2. `pressureSide`
3. `turnArc`
4. `highlightReason`
5. 用户 / AI move tag
6. 上一版 StoryState 继承

### 7.5 为什么这样收
- 能保证第一版稳定、可测、可回归；
- 不把连续性机制再变成一个新的不稳定模型调用点；
- 也足够支撑后续 Tasks 做明确拆解。

## 8. Provider 参数档位映射（推荐版本）
### 8.1 当前现实约束
当前 admin 模型配置已具备：
- `modelName`
- `baseUrl`
- `apiKey`
- `thinkingLevel`
- `enabled`

当前 plan 不引入新的复杂参数面板，优先在既有配置能力内完成 V2 落地。

### 8.2 第一版推荐映射
- **普通回合**：默认走 narrative 配置下的高质量档，不再默认视为轻量路径
- **关键回合**：沿用同一模型族，但允许在请求侧提高表现档位 / 允许更长输出预算 / 更宽 timeout
- **特殊事件**：同样沿用 narrative 配置，但允许再提升事件级档位

### 8.3 thinkingLevel 的使用建议
当前建议先用已有 `thinkingLevel` 做有限映射，而不是扩数据库字段：
- `normal`：普通回合默认档
- `high`：关键回合 / 特殊事件增强档
- 若未来确实需要普通 / 关键 / 事件三档完全独立，再考虑扩配置结构

### 8.4 timeout 的建议映射
- 普通回合：允许比旧策略更宽，但仍需明确超时边界
- 关键回合：在普通回合基础上适度上浮
- 特殊事件：可再上浮，但不能无限等待

### 8.5 当前取舍
这一版先不做：
- 多模型供应商分流
- 普通 / 高光 / 特殊事件三套独立数据库配置
- 后台可视化档位编排器

原因是这些都属于下一层复杂度，不是当前进入 Tasks 的前提。

## 9. 实施路径（MVP 路径）
1. **先扩 shared 契约与类型**
   - 在 `packages/shared/src/index.ts` 中引入 Narrative V2 所需类型、StoryState 类型与新展示模型
2. **再做 server 侧 Narrative V2 主链**
   - 在 `narrative-service.ts` 中引入新的 request / response 组装与解析逻辑
   - 在 `narrative-fallback.ts` 中引入 fallback-v2
3. **再做 web 侧最小新展示层**
   - 先在 `presentation.ts` / `App.tsx` 中跑通新结构显示
4. **再补特殊事件结构与移动端细化**
   - 把事件结构与弱事实层补齐
5. **最后收成本策略与参数映射**
   - 让普通回合高投入与 fallback 边界都稳定下来

### 最小可演示切片
最小可演示切片建议为：
- 普通合法回合一条完整新链路
- 含 3 个角色的对戏主体
- 1 句收束
- 1 行弱事实尾注
- `StoryState` 成功跨至少 3 回合持续更新
- 前端已按新结构展示，不再显示旧四段式标签

### 推荐拆分顺序（便于后续 Tasks）
- Slice A：shared 类型与 StoryState 结构扩展
- Slice B：server 普通回合 Narrative V2
- Slice C：web 普通回合新展示层
- Slice D：server / web 特殊事件 V2
- Slice E：成本策略、fallback 与展示细化

### 更贴近执行的任务组建议
- Task Group P-1：shared 契约与 StoryState 更新逻辑
- Task Group P-2：server 普通回合 Narrative V2 生成链
- Task Group P-3：web 普通回合新展示层
- Task Group P-4：特殊事件结构迁移
- Task Group P-5：成本 / timeout / fallback 收稳

## 10. 风险与取舍
### 主要风险
- 新演绎生成质量不稳定，导致“更自由但更飘”
- `StoryState` 设计不当，反而增加复杂度却没有明显提升连续戏感
- 前端显示改造不彻底，旧四段式结构阴影残留
- 成本上限提高后，响应时间和失败边界若没处理好，会影响主链路体验
- 现有 shared 单文件入口过大，若直接堆太多新类型，后续维护会变重

### 主要取舍
- 为什么采用当前方案：
  - 因为当前问题是产品结构错位，不是局部润色问题；
  - 必须重做演绎可见层和连续性层，才能真正对准目标体验。
- 为什么不采用更复杂方案：
  - 当前不引入第三个 AI 摘要器、不做后台剧情编辑器、不做多世界观系统；
  - 先把最关键的普通回合体验与连续戏感做实，再考虑复杂扩展。
- 为什么不先做大规模文件重构：
  - 当前真实承载点很集中，先在原位跑通 MVP 更快；
  - 等 V2 成立后，再做结构化拆分更稳。

## 11. 复杂度与简化说明
### 本轮保留的必要复杂度
- `StoryState` 结构化连续性机制
- Narrative Contract V2
- 普通回合与特殊事件分结构处理
- 移动端分层展示策略
- 高投入模型调用 + 边界型 fallback

### 本轮刻意不引入的复杂度
- 第三个轻量 AI summarizer
- 多主题 / 多世界观切换
- 角色编辑后台
- 用户可调演绎强度面板
- 多供应商混编调度
- 第一阶段就全面拆文件 / 重写展示架构

## 12. 验证策略
### 主链路验证
- 合法落子后，普通回合应返回新的 Narrative V2 结构
- 前端不再展示旧四段式标签
- 连续 3～5 回合后，`StoryState` 能持续更新且表现出连续戏感
- 移动端能看到常驻单行弱事实摘要，并可展开详情

### 用户故事独立验证
- 用户故事 1 → 连续数回合的主体验是“看人物对戏”而非“看结构化解说”
- 用户故事 2 → 至少 3 类角色的语言风格能被辨认
- 用户故事 3 → 连续回合能感知局势推进与张力变化
