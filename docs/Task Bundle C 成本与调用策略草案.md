# Task Bundle C 成本与调用策略草案

## 文档定位
- 文档类型：成本与调用策略草案
- 适用范围：Task Bundle C AI 演绎升级（V1）
- 当前状态：草案
- 最后更新时间：2026-03-27

## 1. 文档目的
本文件用于定义：
- AI 演绎能力在 V1 中如何分层调用
- 普通回合与高光事件应如何分配预算
- 调用失败、超时、限流时如何降级
- 如何在“亮点能力”与“可持续成本”之间取得平衡

## 2. 基本判断
当前已确认：
- 演绎是本项目的最大亮点之一
- 为了演绎效果，可以接受比普通功能更高的 AI 成本
- 但不建议“无节制地贵”，而应采用可控、分层、可降级的成本策略

因此 V1 的原则不是“所有回合都用最贵模型”，而是：

> **把预算优先花在真正影响体验识别度的地方。**

## 3. 分层调用原则
### 3.1 模型族策略
当前已确认：
- 普通回合与高光事件先采用同一供应商 / 同一模型族
- 第一版不通过更换供应商或切换完全不同模型族来制造差异

### 3.2 普通回合
普通回合建议通过以下维度与高光事件拉开差异：
- 字数更短
- 语气更克制
- 角色表现更收敛
- 更强调节奏稳定与事实贴合

### 3.3 特殊事件 / 高光回合
以下情况建议允许更高预算或更强表现：
- 非法步
- 悔棋
- 将军
- 终局
- 明显反杀
- 局势突变

但当前建议高光与普通的主要差异，先体现在：
- 字数
- 演绎语气
- 角色表现强度

### 3.4 不建议的做法
- 不建议为了早期效果，立刻把普通与高光拆成完全不同供应商或不同模型族
- 不建议为了“显得聪明”让普通回合输出过长
- 不建议把预算主要花在冗长文案，而不是更稳的结构化输出上

## 4. 推荐策略（V1）
### 4.1 推荐默认方案
V1 推荐采用：
- 普通回合与高光事件：同一供应商 / 同一模型族
- 差异主要通过字数、语气、角色表现强度与提示词约束拉开
- 所有调用都受统一 schema 约束
- 失败统一 fallback

### 4.2 为什么推荐这个方案
它兼顾了：
- 成本可控
- 普通体验稳定
- 高光时刻更有表现力
- 后续可扩展

## 5. 事件分级建议
### 5.1 L1：普通回合
示例：
- 无吃子
- 无将军
- 无终局
- 普通推进 / 抢位 / 试探 / 解围

建议：
- 使用轻量模型
- 严格控制输出字数
- 输出以“清晰 + 有角色感”为主

### 5.2 L2：明显关键回合
示例：
- 吃子
- 关键护驾
- 明显施压
- 局势开始剧烈倾斜

建议：
- 允许提升推理或表现档位
- 但仍保持结构化输出

### 5.3 L3：特殊事件 / 高光回合
示例：
- 非法步
- 悔棋
- 将军
- 终局
- 明显反杀 / 翻盘节点

建议：
- 使用增强模型或增强调用档位
- 优先保证张力、辨识度与剧情收束感

## 6. 调用流程建议
### 6.1 普通流程
1. 系统生成结构化叙事输入
2. 依据事件分级选择模型档位
3. 调用 AI 获取结构化输出
4. 做 schema 校验
5. 进入统一时间线

### 6.2 失败处理流程
1. 若超时 / 非 JSON / schema 不合法 / 内容脱离事实
2. 立即降级到 fallback
3. 仍写入统一时间线
4. 不影响棋局主链路

## 7. 超时与降级策略
### 7.1 普通回合
- 应更激进地超时降级
- 原则：宁可更普通，也不能拖慢对局主节奏

### 7.2 特殊事件 / 高光回合
- 可允许略高等待上限
- 但仍必须有明确超时边界
- 不允许无限等待，避免影响用户操作闭环

### 7.3 统一降级原则
- 对用户来说，降级后应仍有可读内容
- 不允许因为 AI 失败导致该回合 / 该事件完全“消失”
- 降级后可少一点戏剧感，但不能失真

## 8. 缓存策略建议
### 8.1 可以考虑缓存的内容
- 同一局内已生成的普通回合演绎结果
- 已生成的特殊事件结果
- 剧情圣经与角色卡等静态材料

### 8.2 不建议缓存的内容
- 跨局直接复用完整演绎正文
- 跨主题直接复用
- 未区分事实输入就复用高光事件正文

### 8.3 缓存原则
缓存是为了省成本与稳响应，不是为了制造模板感。

## 9. 成本优先级建议
如果预算有限，优先级建议为：
1. 特殊事件（非法步 / 悔棋 / 将军 / 终局）
2. 高光回合
3. 普通回合

如果预算充足，可提升普通回合质量，但仍建议保留事件分级机制。

## 10. 质量优先级建议
如果效果与成本冲突，优先级建议为：
1. 事实贴合
2. 统一时间线不断裂
3. 特殊事件表现力
4. 普通回合角色感
5. 普通回合辞藻丰富度

也就是说：
- 不要为了文采牺牲事实
- 不要为了高光文案牺牲主链路响应

## 11. 运维与配置建议
### 11.1 V1 建议
- 不把调用策略完全交给用户前台配置
- 先作为系统级或后台级策略控制
- 前台不暴露“演绎强度”调参

### 11.2 可保留的后台策略项（后续再确认）
- 普通回合模型档位
- 特殊事件模型档位
- 超时阈值
- fallback 开关与日志记录

## 12. 测试建议
### 12.1 成本策略测试
至少应覆盖：
- 普通回合走轻量路径
- 高光事件走增强路径
- 超时后正确 fallback
- schema 非法后正确 fallback

### 12.2 体验测试
至少应覆盖：
- 普通回合响应不明显拖慢主链路
- 特殊事件文本明显优于普通回合
- 降级时用户仍能看到完整时间线

## 13. 本轮已确认事项
1. 普通回合与高光事件采用同一供应商 / 同一模型族
2. 高光事件的判定阈值这一版不再继续细分
3. 不对连续多回合演绎做成本上限控制
4. 记录 AI 调用失败原因，以便后续调优
5. 不对同一局设置总预算上限

## 14. 当前待确认事项
1. 在同一模型族内，普通回合与高光事件具体如何通过提示词、字数和表现强度拉开差异
2. 是否需要记录“失败原因 + 回合类型 + 事件类型”的组合日志，便于后续精调
3. 若后续成本明显高于预期，优先先收哪一侧：普通回合长度、角色数量，还是高光事件表现强度

## 15. 当前结论
本草案的核心结论是：

> Task Bundle C 的 AI 演绎能力应采用“普通回合轻量、特殊事件增强、统一 schema、失败可降级”的分层调用策略，把预算集中投向真正影响产品辨识度的高光时刻。

本文件当前仍为草案，待后续进一步确认后，再进入实现与后台策略设计。
