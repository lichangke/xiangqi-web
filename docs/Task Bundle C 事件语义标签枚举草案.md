# Task Bundle C 事件语义标签枚举草案

## 文档定位
- 文档类型：事件语义标签草案
- 适用范围：Task Bundle C AI 演绎升级（V1）
- 当前状态：草案
- 最后更新时间：2026-03-27

## 1. 文档目的
本文件用于把 V1 中特殊事件使用的 `eventSemanticTag` 收敛成一套固定枚举表。

它主要回答以下问题：
1. 特殊事件侧到底允许哪些语义标签
2. 这些标签分别适用于哪些事件类型
3. 哪些标签当前不建议纳入第一版
4. 系统在构造特殊事件输入时，应如何稳定选择这些标签

## 2. 总体结论
当前建议：

> V1 的 `eventSemanticTag` 应使用固定枚举，而不是让演绎 AI 或系统在每次事件发生时临时自由命名。

原因有三个：
- 便于测试与校验
- 便于演绎 AI 稳定消费
- 便于保持事件文案风格一致，不至于同类事件反复换说法

## 3. 与普通回合标签的边界
`eventSemanticTag` 与普通回合中的：
- `userMoveTag`
- `aiMoveTag`
- `turnArc`

不是同一层东西。

### 普通回合标签解决的是：
- 这一手像什么
- 这一来一回推进到什么阶段

### 事件语义标签解决的是：
- 这个特殊事件在剧情上该被归成什么性质

因此当前建议：
- 不把普通回合标签和事件标签混成一套大杂烩
- 单独为特殊事件维护一组更稳定、更克制的固定标签

## 4. 第一版推荐固定枚举
当前建议 V1 只保留以下 6 个事件语义标签：

1. `军议否决`
2. `推演回拨`
3. `正面施压`
4. `主动收局`
5. `胜负已定`
6. `规则裁断`

## 5. 各标签定义与适用范围

## 5.1 军议否决
### 含义
表示某个尝试方案未被允许进入正式执行，带有“当场否决”的意味。

### 推荐适用事件
- `illegal_move`

### 推荐语感
- 克制
- 明确
- 带一点裁定感
- 不写成情绪化指责

### 不建议适用
- `undo`
- `finish`

因为这两个事件不是“方案被否”，而是“局势回拨”或“结果落定”。

## 5.2 推演回拨
### 含义
表示已有推演被回退到前一判断点，重点在“撤回上一轮推演”而不是简单撤销按钮动作。

### 推荐适用事件
- `undo`

### 推荐语感
- 不要写成系统提示腔
- 要体现“局势回拨”而不是“操作成功”

### 不建议适用
- `illegal_move`
- `resign`

## 5.3 正面施压
### 含义
表示局势已进入明确压迫状态，对方必须立刻应对。

### 推荐适用事件
- `check`

### 推荐语感
- 紧迫
- 有压迫感
- 但不等于终局收束

### 不建议适用
- `finish`

因为将军只是高压节点，不等于结果已经定下。

## 5.4 主动收局
### 含义
表示其中一方主动结束对局，重点是“收局意志”而不是“失败播报”。

### 推荐适用事件
- `resign`

### 推荐语感
- 克制
- 有收束感
- 不写成羞辱式失败文案

### 不建议适用
- `finish`

因为 `finish` 更偏系统最终结论，而 `resign` 更偏主动选择结束。

## 5.5 胜负已定
### 含义
表示对局已经进入明确结果态，重点在“结果被坐实”。

### 推荐适用事件
- `finish`

### 推荐语感
- 收束
- 定局
- 不宜再写成“仍在缠斗中”

### 推荐说明
该标签适合：
- `checkmate`
- `resign`
- `rule_settlement`

对应的 finish 场景。

## 5.6 规则裁断
### 含义
表示结果或处理方式来自规则系统的正式裁断，而不是某一方单步主动触发。

### 推荐适用事件
- `finish`（当 `finishReason=rule_settlement` 时优先）
- 后续若新增 `draw` 等规则结算事件，也可复用

### 推荐语感
- 正式
- 克制
- 有裁定感

### 与“胜负已定”的区别
- `胜负已定` 强调结果已经明确
- `规则裁断` 强调结果来自规则层面的归结与认定

## 6. 事件类型到标签的默认映射
当前建议 V1 默认映射如下：

| eventType | 默认 eventSemanticTag |
|---|---|
| `illegal_move` | `军议否决` |
| `undo` | `推演回拨` |
| `check` | `正面施压` |
| `resign` | `主动收局` |
| `finish` + `finishReason=checkmate` | `胜负已定` |
| `finish` + `finishReason=resign` | `胜负已定` |
| `finish` + `finishReason=rule_settlement` | `规则裁断` |

## 7. 第一版不建议纳入的标签
以下表达当前建议先不要纳入固定枚举：
- `军心动摇`
- `局势倾覆`
- `阵线崩解`
- `裁决落槌`
- `封盘定论`
- `撤案重议`
- `锋线压顶`

原因不是这些词不能写，而是：
- 太文学化
- 与现有标签边界容易重叠
- 第一版不利于稳定测试与统一消费

这些表达更适合出现在最终文案层，而不是固定标签层。

## 8. 系统选择规则建议
当前建议系统选择 `eventSemanticTag` 时，优先走固定映射：

1. 先看 `eventType`
2. 若 `eventType=finish`，再看 `finishReason`
3. 直接映射到固定标签
4. 不让演绎 AI 自由改写标签名称

### 例外情况
第一版原则上不开放例外。

也就是说：
- 相同事件类型，优先保持同一标签
- 不因文案想写得更“新鲜”而频繁换标签名

## 9. 演绎 AI 消费约束
演绎 AI 在消费 `eventSemanticTag` 时：
- 可以把标签翻译成不同措辞的最终文案
- 但不应反向改写标签含义
- 不应把 `军议否决` 写成像 `推演回拨`
- 不应把 `主动收局` 写成像 `胜负已定`

也就是说：
- 文案可以灵活
- 标签语义必须稳定

## 10. 测试建议
至少应覆盖：
1. 同一事件类型是否稳定命中同一标签
2. `finishReason` 不同是否正确分流到 `胜负已定 / 规则裁断`
3. 演绎 AI 是否真正根据标签区分文案气质
4. fallback 时是否仍能保留正确标签语义

## 11. 当前待确认事项
1. `finishReason=resign` 时，是否长期固定映射为 `胜负已定`，还是后续允许用 `主动收局` 覆盖
2. 后续若新增 `draw`，是否直接归入 `规则裁断`
3. 是否需要把 `eventSemanticTag` 同步回写进《特殊事件统一 Schema 草案》的示例与待确认项

## 12. 当前结论
本草案的核心结论是：

> V1 的 `eventSemanticTag` 应收敛为少量固定枚举，并通过“事件类型 → 语义标签”的稳定映射，为特殊事件演绎提供统一、可测、可控的解释层。

本文件当前仍为草案，待后续进一步确认后，再决定是否并入《Task Bundle C 特殊事件统一 Schema 草案》与正式执行文档。
