# 需求分析结论

## 1. 项目定位
本项目是一个以中国象棋为核心的网页版产品。

V1 核心不是单纯“能下棋”，而是同时具备以下特征：
- 严格符合中国象棋规则的人机对战
- AI 具备棋子人格与话剧式讨论演绎
- 对局过程可玩、可看、可点评
- 支持电脑端与手机端访问

---

## 2. V1 当前明确包含
- 人机对战
- 严格规则校验
- 非法落子提示（可风趣、幽默、轻讽刺，但不应令人反感）
- 悔棋
- 认输
- AI 棋子讨论区 / 回合展示区
- 对局结束后的 AI 简短评价，且风格需与整局演绎统一
- 普通用户登录
- 后台管理页面
- 大模型相关配置管理
- 3 种页面主题
- 支持电脑端与手机端使用

---

## 3. V1 当前明确不做，但架构需预留
- 人人对战
- 房间 / 匹配 / 邀请
- 联机实时同步
- 观战模式
- 棋局保存与恢复的正式实现
- Qt 桌面客户端本体

说明：
- 虽然 V1 暂不实现人人对战，但架构需为后续联机扩展预留空间。
- 虽然 V1 暂不实现 Qt 桌面客户端，但本轮架构设计需避免将核心逻辑写死在 Web 页面层。

---

## 4. 已确认的关键边界

### 4.1 多端支持
- 同一套 Web 产品需支持电脑端与手机端访问。
- 主链路在移动端也必须可用，但布局与交互可与桌面端不同。

### 4.2 用户登录的当前意义
保留普通用户登录，但当前意义先不是联机，而是：
- 用户身份
- 偏好配置
- 后续存档能力预留

### 4.3 Qt 预留原则
本轮架构设计需考虑未来接入 Qt 桌面客户端，原则上应做到：
- 棋规引擎独立于具体 UI
- AI 演绎逻辑独立于具体 UI
- 用户 / 配置 / 对局数据访问边界清晰
- Web 只是一个客户端形态，而不是核心逻辑唯一承载层

### 4.4 悔棋规则
悔棋规则已确认如下：
- 每次悔棋 = 撤销最近一个完整回合（玩家一步 + AI 一步）
- 悔棋后不能立刻再次悔棋，必须先完成一次新的正常落子后，才能再次悔棋
- 每局最多 5 次悔棋机会

### 4.5 并发策略
当前倾向已确认：
- 不先限制总在线人数
- 重点限制 AI 对局并发
- 限制单账号同时进行中的对局数

### 4.6 注册策略
注册策略已确认如下：
- V1 默认关闭公网自由注册
- 普通用户由后台创建
- 架构预留邀请码注册能力
- 未来可按运营需要开启邀请码注册

### 4.7 手机端交互主策略
手机端交互当前已确认如下：
- 竖屏可用，横屏体验更佳
- 棋盘为手机端视觉中心
- 讨论区默认折叠，以抽屉 / Tab 方式展开
- V1 采用“点击选子 + 点击目标点”落子，不做拖拽落子
- 非法落子提示采用轻量风格化提示，不使用重模态长期遮挡棋盘
- 横屏时允许讨论区常驻在侧边
- AI 思考中，讨论区未展开时可仅显示摘要，展开后查看完整内容
- 主题切换放入设置中，而不是主操作区常驻

### 4.8 普通用户在 V1 的具体价值落点
V1 普通用户的核心价值先限定为：身份识别、偏好保存、基础对局记录，以及为后续棋局存档恢复预留数据锚点。

具体包括：
- 账号身份：用户名、密码登录、基础状态（启用 / 禁用）
- 偏好配置：主题偏好、棋盘朝向 / 显示偏好、是否默认展开讨论区，以及必要时保留简单的 AI 演绎风格偏好
- 基础对局记录：最近对局列表、对局结果、开始 / 结束时间、所选难度、是否认输结束、悔棋使用次数
- 扩展预留：为后续棋局保存与恢复、人人对战、Qt 客户端接入预留用户身份锚点

当前明确不强行纳入：
- 好友系统
- 排行榜
- 成就系统
- 复杂战绩统计
- 云端完整棋谱管理
- 多终端实时同步状态

### 4.9 后台管理边界
V1 后台管理边界限定为：用户管理、模型配置管理、运行策略配置，以及轻量审计与观察能力；不扩展到复杂运营平台和高级权限系统。

具体包括：
- 用户管理：创建用户、启用 / 禁用用户、重置密码、查看基础信息，不提供明文密码查看
- 模型配置管理：模型名称、API Key、Base URL、思考强度、连通性测试、启用 / 停用配置；必要时支持区分“对局决策模型”和“演绎模型”是否同一套
- 运行策略配置：AI 对局并发上限、单账号同时进行中的对局数上限、注册策略开关、系统默认值，以及必要时预留悔棋次数上限等可配置项
- 轻量审计与观察：谁创建用户、谁重置密码、谁修改模型配置、谁改动关键系统策略、模型连通性测试结果摘要

当前明确不纳入：
- 复杂 BI 报表
- 详细运营分析大盘
- 高级权限系统
- 细粒度 RBAC
- 复杂告警中心
- 全量日志检索平台

## 5. 规则口径当前共识

### 5.1 规则总原则
V1 采用“硬规则引擎裁判制”：
- 人类落子是否合法，由规则引擎判定
- AI 的可选走法范围先由规则引擎生成
- 大模型不能绕过规则直接落子
- AI 的价值主要放在策略表达与戏剧演绎，而不是合法性判断

### 5.2 V1 规则目标口径
- V1 以正式竞赛规则为目标口径
- 主链路需严格覆盖：
  - 各棋子标准走法
  - 马腿、象眼、炮架
  - 九宫 / 河界限制
  - 将军判定
  - 将帅不能照面
  - 自陷被将的非法步判定
  - 将死判负
  - 无合法走法时的终局判定
  - 认输判负
  - 悔棋规则

### 5.3 规则附录的权威依据
- 规则附录统一以《中国象棋竞赛规则》的现行正式版本作为唯一权威依据
- 长将、长捉、循环局面、和棋判定与特殊终局裁定口径，后续均按该依据查证并落盘

### 5.4 后续需专项查证并落盘的规则附录项
以下细则后续需形成专项规则附录：
- 长将
- 长捉
- 循环局面
- 和棋判定
- 特殊终局裁定口径

### 5.5 规则层实现策略
- V1 规则层优先评估并复用成熟 JS 规则库作为 baseline，首选评估 xiangqi.js
- 第三方规则库必须通过自定义适配层接入，不能让业务层直接散落依赖其原生 API
- 最终裁判依据仍是《中国象棋竞赛规则》与后续专项测试集，而不是第三方库的默认行为

### 5.6 当前补充倾向
- V1 暂不引入读秒 / 用时制，除非后续明确纳入范围

## 6. AI 难度机制当前共识

### 6.1 总体原则
- 前台只暴露一个“难度”选择器
- 难度主要由策略层控制，而不是简单等同于“换更大的模型”
- 模型主要负责：
  - 角色讨论
  - 风格表达
  - 对候选走法做带人格的解释
- 最终落子必须受合法走法集合硬约束

### 6.2 V1 前台难度档位
前台先定为 4 档：
- 新手
- 普通
- 困难
- 大师

### 6.3 演绎强度
- V1 不把“演绎强度”作为前台用户可调项
- 演绎强度先作为后台或系统配置项处理

### 6.4 补充原则
- AI 的人格表演不能破坏棋力逻辑
- 不能为了戏剧效果输出非法落子或明显越界的落子结论

### 6.5 AI 演绎展示规则
V1 展示的是“角色化、戏剧化、可公开的演绎讨论”，而不是大模型原始完整思维链。

当前展示规则已确认如下：
- 每个有效回合统一以“第 N 回合”开头
- 每回合默认采用统一结构：
  1. 对用户这步的简短点评
  2. 2～4 个角色的短发言
  3. 1 句共识 / 拍板
  4. AI 最终落子 + 1 句简短理由
- 每回合默认控制在约 120～180 字以内
- 展示效果采用“逐段出现”的演出感，可先显示“众将商议中”等状态，再逐条补上角色发言，最后显示最终落子
- 非法落子不新增回合编号，仅给轻量风格化提示
- 悔棋、认输、将军、终局等特殊事件使用独立事件模板
- 每局固定一个演绎主题，局内不切换主题；终局评价需延续同一演绎风格
- 所有棋子角色都应有出场机会，包括：兵、马、炮、车、士、象、帅
- “所有角色有出场机会”指整局或整体产品体验中应为各类角色提供合理出场空间，不要求每个回合全员发言
- 每个回合按局面相关性选择出场角色，不宜每回合全员齐出，以避免内容冗长、成本失控与节奏混乱
