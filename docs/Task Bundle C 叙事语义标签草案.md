# Task Bundle C 叙事语义标签草案

## 文档定位
- 文档类型：语义标签草案
- 适用范围：Task Bundle C AI 演绎升级（V1）
- 当前状态：草案
- 最后更新时间：2026-03-27

## 1. 文档目的
本文件用于定义：
- 决策 AI 在生成机器落子时，应该顺带产出哪些叙事语义标签
- 这些标签的第一版固定枚举如何收敛
- 这些标签如何作为演绎 AI 的上游输入，而不是由演绎 AI 自由猜测

它的目标不是让标签系统过度复杂，而是让演绎 AI 拿到足够稳定、足够可控的“这一手在剧情上更像什么”的解释。

## 2. 总体原则
### 2.1 标签由决策 AI 产出，不由演绎 AI 反推
当前建议：
- 用户落子后的语义解释，可由决策 AI 基于棋盘全局顺带判断
- 机器落子的语义解释，也由决策 AI 在生成落子时同步给出
- 演绎 AI 只消费标签，不重新自由推断标签

### 2.2 第一版追求“够用”，不追求过细
如果标签枚举过多，会带来三个问题：
- 决策 AI 输出不稳定
- 演绎 AI 消费难度上升
- 前期测试与调优成本明显上升

因此 V1 应保持：
- 标签数量有限
- 含义边界相对清晰
- 允许少量语义重叠，但不允许大量近义枚举并存

### 2.3 标签服务于演绎，不替代棋局事实
标签不能替代：
- 落子坐标
- 是否吃子
- 是否将军
- 是否终局
- 是否非法步
- 是否悔棋

标签只是对“这一步在剧情上像什么”的受控解释层。

## 3. 第一版标签层级建议
V1 建议只保留三层：

### 第一层：单步语义标签
描述某一方这一手更像什么。

### 第二层：回合推进标签
描述这一来一回之后，整回合在剧情上进入了什么阶段。

### 第三层：高光原因标签
描述为什么这回合值得比普通回合更强烈地写。

## 4. 单步语义标签建议枚举
以下标签建议同时适用于：
- `userMoveTag`
- `aiMoveTag`

## 4.1 推荐保留的基础标签
1. `试探`
   - 用于轻量试边、摸底、观察回应
   - 典型语感：先探一步，不急于摊牌

2. `抢位`
   - 用于争夺关键位置、线路或节奏点
   - 典型语感：不是直接杀伤，但在抢主动权

3. `压迫`
   - 用于持续施压、逼迫对方应手
   - 典型语感：开始把对手往被动方向推

4. `解围`
   - 用于化解眼前压力、拆除迫近风险
   - 典型语感：先止血，再谈反推

5. `护驾`
   - 用于保护中枢、稳住帅/将周边安全
   - 典型语感：不是保守，而是先稳底线

6. `换子`
   - 用于主动或被动交换关键子力
   - 典型语感：通过交换改变局势结构

7. `佯动`
   - 用于制造假目标、引诱应手、声东击西
   - 典型语感：表面如此，实则另有所图

8. `追击`
   - 用于延续上一回合已形成的追压或逼退
   - 典型语感：不是新开线，而是把已有优势继续压深

9. `收束`
   - 用于局势已经趋向明确，开始进入收官或封口
   - 典型语感：不再铺陈，而是把局面往结果推

## 4.2 当前不建议第一版保留的标签
以下标签当前建议先不单独立枚举：
- `奇袭`
- `围杀`
- `试杀`
- `诱杀`
- `腾挪`
- `弃子`
- `封锁`
- `反制`

原因不是它们没价值，而是第一版太容易与现有标签重叠。

例如：
- `奇袭` 常可并入 `压迫` 或 `佯动`
- `反制` 常可并入 `解围` 或 `追击`
- `封锁` 常可并入 `抢位` 或 `压迫`

## 5. 回合推进标签建议枚举
`turnArc` 用于描述“这一来一回之后，整回合剧情推进到哪里了”。

V1 建议控制在以下范围：
1. `试探延续`
2. `开始抢势`
3. `压力升级`
4. `局面回稳`
5. `攻守换边`
6. `进入收束`

### 使用原则
- `turnArc` 是整回合级别，不是单步级别
- 每回合最多一个
- 它主要服务于演绎 AI 的段落主轴与 summary 生成

## 6. 高光原因标签建议枚举
`highlightReason` 用于解释为什么某回合值得比普通回合更强地表现。

V1 建议保留：
1. `吃子`
2. `将军`
3. `解将`
4. `关键换子`
5. `局势逆转`
6. `终局临近`
7. `特殊事件`

### 使用原则
- 普通回合可为空
- 一回合最多 1～2 个
- 该字段主要服务于：
  - 是否提升表现强度
  - 是否增加一点意象词/角色张力
  - 是否提高该条时间线项的视觉权重

## 7. 标签组合建议
### 7.1 普通回合
建议结构：
- `userMoveTag`：1 个
- `aiMoveTag`：1 个
- `turnArc`：1 个
- `highlightReason`：0 个或 1 个

### 7.2 高光回合
建议结构：
- `userMoveTag`：1 个
- `aiMoveTag`：1 个
- `turnArc`：1 个
- `highlightReason`：1 个或 2 个

### 7.3 不建议
- 单步同时给 2 个以上语义标签
- 回合推进标签一次给 2 个
- 用自然语言长句替代固定枚举

## 8. 输出示例
### 8.1 普通回合示例
```json
{
  "userMoveTag": "试探",
  "aiMoveTag": "压迫",
  "turnArc": "压力升级",
  "highlightReason": []
}
```

### 8.2 高光回合示例
```json
{
  "userMoveTag": "解围",
  "aiMoveTag": "追击",
  "turnArc": "攻守换边",
  "highlightReason": ["将军", "关键换子"]
}
```

## 9. 与特殊事件的关系
特殊事件不一定沿用普通回合的 `userMoveTag / aiMoveTag` 结构，但仍建议保留一个更轻量的事件语义字段，例如：
- 非法步：`军议否决`
- 悔棋：`推演回拨`
- 认输：`主动收局`
- 终局：`胜负已定`

这样可以保持：
- 普通回合走“单步标签 + 回合推进标签”
- 特殊事件走“事件语义标签”

两者虽然字段不同，但都属于“结构化叙事解释层”。

## 10. 测试建议
至少应覆盖：
1. 决策 AI 是否能稳定从固定枚举中输出标签
2. 同一类局势下标签是否明显漂移
3. 演绎 AI 是否真正消费了标签，而不是忽略标签自由发挥
4. 标签是否明显提升了剧情连续性与一致性

## 11. 当前待确认事项
1. `试探 / 抢位 / 压迫 / 解围 / 护驾 / 换子 / 佯动 / 追击 / 收束` 是否作为 V1 的最终单步标签全集
2. `turnArc` 是否保留 6 个枚举，还是进一步压缩到 4 个
3. `highlightReason` 是否允许同回合返回 2 个，还是只允许 1 个
4. 特殊事件语义标签是否要与普通回合标签体系彻底分开管理

## 12. 当前结论
本草案的核心结论是：

> Task Bundle C 的 semanticTag 第一版不应放任演绎 AI 自由理解，而应由决策 AI 在固定枚举内顺带产出，并通过“单步标签 + 回合推进标签 + 高光原因标签”三层结构，把棋局语义稳定地交给演绎 AI 消费。

本文件当前仍为草案，待后续进一步确认后，再决定是否并入正式输入输出契约。
