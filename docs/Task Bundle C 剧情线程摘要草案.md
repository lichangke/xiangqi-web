# Task Bundle C 剧情线程摘要草案

## 文档定位
- 文档类型：剧情线程摘要草案
- 适用范围：Task Bundle C AI 演绎升级（V1）
- 当前状态：草案
- 最后更新时间：2026-03-27

## 1. 文档目的
本文件用于定义 `storyThreadSummary` 在 V1 中的职责、生成方式与控制边界。

它主要解决的问题不是“再多给 AI 一段背景”，而是：
- 演绎 AI 如何把控整场演绎的逻辑性
- 演绎 AI 如何保持回合之间的连贯性
- 演绎 AI 如何让同一局棋有持续推进的故事感，而不是每回合重新开机

## 2. 总体结论
当前建议：

> `storyThreadSummary` 应作为演绎 AI 的核心连续性输入存在，但它不应是松散长文，而应是一个受控、短小、结构化的“剧情线程状态”。

它的职责不是复述整局棋谱，而是告诉演绎 AI：
- 最近这几回合主要在讲什么
- 当前紧张点在哪里
- 局势是升级了、缓和了，还是换边了
- 哪个角色/阵营目前更值得占据叙事中心

## 3. 为什么需要它
如果演绎 AI 每次只看到：
- 当前回合的走子事实
- 当前回合的 semanticTag
- 当前回合的 situationShift

它仍然可能出现三个问题：
1. 每回合像独立小作文
2. 上一回合的 tension 没有延续
3. 整局缺乏“正在推进同一条故事线”的感觉

因此第一版即使不做复杂长篇剧情，也仍需要一个最小连续性锚点。

## 4. 设计原则

## 4.1 要短，不要长
`storyThreadSummary` 不是把最近几回合全文拼接起来。

当前建议：
- 默认控制在 80～150 字
- 极端情况下不超过 200 字
- 优先保留“主线矛盾”和“当前局面阶段”，而不是保留细碎辞藻

## 4.2 要结构化，不要散文化
它不应该是任意自然语言散文，而应尽量对应几个固定槽位。

## 4.3 要讲“局势故事”，不要讲“所有细节”
它关注的是：
- 当前谁在施压
- 这几回合的主线是什么
- 当前回合应延续还是扭转什么

而不是逐条复述每一步坐标。

## 4.4 它服务于演绎 AI，不服务于规则系统
规则判定、合法性、将军、终局、悔棋等，仍由系统事实层负责。

`storyThreadSummary` 只影响：
- 演绎是否连贯
- 角色出场是否连续
- 整局是否有统一故事推进感

## 5. V1 推荐结构
当前建议 `storyThreadSummary` 至少包含以下 5 个槽位：

```json
{
  "currentPhase": "string",
  "mainConflict": "string",
  "pressureSide": "RED | BLACK | BALANCED",
  "recentFocus": "string",
  "carryForward": "string"
}
```

## 5.1 currentPhase
表示当前整局更接近哪个叙事阶段。

建议枚举：
- `试探期`
- `抢势期`
- `对压期`
- `解围期`
- `收束期`

## 5.2 mainConflict
一句短摘要，描述最近几回合真正的主线矛盾。

示例：
- `黑方持续从中路施压，红方暂以护驾和换位稳局`
- `红方刚解掉一波正面压迫，但主动权尚未完全拿回`

## 5.3 pressureSide
表示当前叙事上谁更占主动或更在施压。

建议取值：
- `RED`
- `BLACK`
- `BALANCED`

## 5.4 recentFocus
表示最近 2～3 回合最值得延续的焦点。

示例：
- `中路压力`
- `边线试探`
- `护驾与解围`
- `换子后的重新布势`

## 5.5 carryForward
表示下一回合演绎最应该延续的点。

示例：
- `继续强调黑方压迫感，不要把局势写平`
- `红方虽解围成功，但不要写成已完全翻盘`
- `本轮若再换子，应延续收束感`

## 6. 生成方式建议

## 6.1 当前推荐方式
V1 推荐把 `storyThreadSummary` 视为一个**系统维护的轻量状态对象**，而不是每次都重新从零生成全文摘要。

建议流程：
1. 当前回合结束后，系统拿到：
   - 当前事实结果
   - 决策 AI 给出的 `semanticTag / turnArc / highlightReason / situationShift`
   - 上一版 `storyThreadSummary`
2. 基于这些输入，生成下一版 `storyThreadSummary`
3. 下一回合调用演绎 AI 时，再把该摘要作为连续性输入带上

## 6.2 谁来生成
当前建议有两个可选方案：

### 方案 A：系统规则化更新（优先）
由系统基于固定字段拼装与更新。

优点：
- 可控
- 稳定
- 成本低

缺点：
- 表达可能较硬

### 方案 B：轻量 AI 更新（后续可选）
由一个轻量 summarizer 根据上一版摘要 + 当前回合结构化结果生成新摘要。

优点：
- 连贯性可能更自然

缺点：
- 增加链路复杂度
- 需要额外测试与兜底

### 当前建议
V1 先采用 **方案 A：系统规则化更新**。

原因：
- 当前重点是把连续性机制立住
- 先不要让链路里再多一个不稳定点
- 等双 AI 主链稳定后，再评估是否需要轻量 AI 做摘要润色

## 7. 保留轮数建议
当前建议：
- `storyThreadSummary` 不直接存“最近 N 回合全文”
- 而是始终只保留 **1 份当前线程状态**
- 该状态由最近 2～4 回合不断更新形成

也就是说：
- 对演绎 AI 来说，只拿到“当前有效摘要”
- 不拿全量历史正文

这样做的好处是：
- token 更稳
- 连续性更可控
- 不容易因为历史过长导致主题漂移

## 8. 与其他字段的关系

## 8.1 与 semanticTag 的关系
- `semanticTag` 解决“这一手像什么”
- `storyThreadSummary` 解决“这几回合整体在讲什么”

## 8.2 与 situationShift 的关系
- `situationShift` 解决“这一来一回后局势怎么变了”
- `storyThreadSummary` 解决“这个变化要放回整局哪个主线里理解”

## 8.3 与 turnArc 的关系
- `turnArc` 是当前回合推进标签
- `storyThreadSummary` 是跨回合的延续性状态

## 9. 输入输出示例
### 9.1 当前摘要示例
```json
{
  "currentPhase": "对压期",
  "mainConflict": "黑方持续从中路施压，红方主要通过护驾和换位维持阵型",
  "pressureSide": "BLACK",
  "recentFocus": "中路压力",
  "carryForward": "下一回合若红方仍以解围为主，不要写成已经翻盘"
}
```

### 9.2 更新后摘要示例
假设本回合红方成功解将并换子，则下一版可能变成：

```json
{
  "currentPhase": "解围期",
  "mainConflict": "红方已暂时拆掉正面威胁，但黑方仍保留后续施压空间",
  "pressureSide": "BALANCED",
  "recentFocus": "解围后的重新布势",
  "carryForward": "下一回合重点不在庆功，而在重新判断谁先抢到下一步主动"
}
```

## 10. fallback 建议
若 `storyThreadSummary` 更新失败：
- 不应阻断主链路
- 可退回到上一版摘要
- 若无上一版摘要，则使用系统最小兜底版本，例如：
  - `当前仍处试探期，局面主线尚未定型`
  - `当前双方仍在争夺主动，尚未进入明确收束阶段`

## 11. 测试建议
至少应覆盖：
1. 连续 3～5 回合后，摘要是否仍然稳定短小
2. 局势反转后，摘要是否能及时改写主线
3. 演绎 AI 是否真正使用了 `storyThreadSummary`，而不是忽略它重复写模板
4. 回退到上一版摘要时，是否会出现明显剧情断裂

## 12. 当前待确认事项
1. `currentPhase` 是否保留 5 档，还是进一步压缩到 4 档
2. `carryForward` 是否作为 V1 必填字段
3. V1 是否允许 `storyThreadSummary` 由轻量 AI 辅助润色，还是完全系统生成
4. `recentFocus` 是否应限制为固定枚举，而不是短文本

## 13. 当前结论
本草案的核心结论是：

> V1 的 `storyThreadSummary` 不应是长篇背景补丁，而应是一个由系统维护、持续滚动更新的短小结构化剧情线程状态，用来帮助演绎 AI 在每回合之间保持逻辑性、连贯性与统一故事推进感。

本文件当前仍为草案，待后续进一步确认后，再决定是否并入正式输入输出契约与双 AI 协作链路。
