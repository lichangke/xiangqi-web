## 1. 任务拆解原则
- 按用户故事优先级组织
- 每组任务尽量独立可验证
- 优先保障 MVP 主链路
- 任务应可执行、可验证、可交接
- `tasks.md` 负责任务结构图，不默认等于一次性全部派发给 Coding Agent 的执行颗粒度

---

## 2. 任务总览
### 任务分组
- 任务组 A：基础框架、认证与规则适配层基线
- 任务组 B：核心对局主链路与 AI 回合闭环
- 任务组 C：演绎展示、事件模板与手机端交互
- 任务组 D：用户偏好、最近对局、三主题与跨端限制
- 任务组 E：后台模型配置、运行策略、未配置状态与审计
- 任务组 F：收尾验证、结构固化与文档补位

### 优先级概览
- P1：A、B、C
- P2：D、E
- P3：F

### 独立验证概览
- 任务组 A → 能完成登录、基础项目框架启动，并对规则适配层进行最小合法性验证
- 任务组 B → 能跑通“用户合法落子 → AI 合法应对 → 局面更新”主链路
- 任务组 C → 能看到逐段出现的回合演绎、非法落子提示、悔棋 / 认输 / 终局事件模板，并在手机端可用
- 任务组 D → 能保存主题 / 偏好 / 最近对局，且跨端同账号限制生效
- 任务组 E → 能在后台配置模型、看到未配置状态提示、调整策略并查看审计摘要
- 任务组 F → 能形成端到端验证、规则样例验证与结构文档收尾

## 3. 任务组划分（供 Implement 派发）
### Task Bundle A：基础框架与规则基线
- 目标：搭起 V1 的全栈骨架、认证基础与规则适配层 baseline
- 包含任务：项目目录落地、前后端基础框架、数据库 schema 初稿、用户名密码认证、RuleAdapter 基线接入与典型规则测试
- 不包含任务：完整 AI 演绎、后台管理页、最近对局展示
- 依赖前提：已确认 spec.md 与 plan.md
- 完成判定：可启动应用、完成登录、规则适配层可校验并返回合法走法 / 非法步结论
- 推荐执行顺序：先基础框架，再认证，再规则层

### Task Bundle B：核心对局与 AI 主链路
- 目标：跑通核心人机对战闭环
- 包含任务：GameSession、落子 API、AI 合法候选步选择、对局状态更新、悔棋 / 认输基础支持
- 不包含任务：复杂后台、完整审计、最近对局摘要页面
- 依赖前提：Task Bundle A 完成
- 完成判定：用户能新建一局、合法落子、看到 AI 合法应对，并可认输 / 悔棋
- 推荐执行顺序：先对局服务，再 AI 策略层，再事件闭环

### Task Bundle C：演绎与移动端体验
- 目标：补齐产品差异化体验与移动端主链路可用性
- 包含任务：回合演绎展示、特殊事件模板、三主题切换、手机端布局与交互、轻量提示
- 不包含任务：后台模型配置持久化、审计摘要
- 依赖前提：Task Bundle B 完成
- 完成判定：逐段出现演绎正常、手机端可用、特殊事件模板统一
- 推荐执行顺序：先展示结构，再事件模板，再移动端适配

### Task Bundle D：后台与用户数据闭环
- 目标：补齐管理员能力与用户数据闭环
- 包含任务：偏好保存、最近对局摘要、跨端进行中对局限制、后台模型配置、运行策略、未配置状态、审计摘要
- 不包含任务：联机对战、复杂运营分析
- 依赖前提：Task Bundle A / B 基础稳定
- 完成判定：管理员可配置模型与策略，用户偏好和摘要可保存恢复，未配置状态有明确提示
- 推荐执行顺序：先用户数据，再后台配置，再审计与未配置状态

默认原则：
- 单次 handoff 给 Coding Agent 时，优先按 task bundle 派发
- 不默认把整个 `tasks.md` 一次性全部交给 Coding Agent
- 只有多个 bundle 强相关且可一起独立验收时，才考虑合并派发

## 5. 用户故事 1（P1）任务组
### 目标
完成严格规则约束下的人机对战主链路，并让 AI 在合法候选步集合中完成应对。

### 对应范围
- 用户故事：US1
- FR：FR-004 ~ FR-017
- 模块：Web 前端、GameSession、RuleAdapter、Strategy

### 独立验证方式
用户登录后，能新建一局，完成合法落子，看到 AI 合法落子后的局面更新。

### 任务列表
#### T11【US1】接入规则适配层 baseline 并完成典型规则测试
- 目标：封装规则库，提供统一的合法走法生成、落子校验、终局判定接口
- 涉及范围 / 文件路径：`src/domain/rules/`、`tests/rules/`
- 前置依赖：T01、T02
- 具体动作：评估并接入 xiangqi.js baseline；实现 RuleAdapter；补齐典型规则样例测试
- 完成定义：业务层不直接依赖第三方原生 API；规则接口可用
- 验证方式：马腿、象眼、炮架、将帅照面、自陷将等用例通过

#### T12【US1】实现对局服务与新建对局 API
- 目标：实现 GameSession 创建、局面状态保存与回合推进基础能力
- 涉及范围 / 文件路径：`src/domain/game/`、`src/server/`
- 前置依赖：T11
- 具体动作：设计 GameSession 状态结构；实现新建对局、读取当前对局、落子处理接口
- 完成定义：可创建一局可持续推进的对局
- 验证方式：接口测试通过，局面状态可正确保存 / 读取

#### T13【US1】实现 AI 合法候选步选择闭环
- 目标：在合法候选步集合中，根据难度生成最终落子
- 涉及范围 / 文件路径：`src/domain/ai/strategy/`
- 前置依赖：T11、T12
- 具体动作：基于难度建立最小策略层；保证最终落子来自合法集合
- 完成定义：AI 回合可稳定生成合法落子
- 验证方式：多组对局样例中 AI 不出现非法步

#### T14【US1】实现用户端基础棋盘页面与对局主链路
- 目标：让用户能在前端完成新建对局、选择棋子、落子和接收回合更新
- 涉及范围 / 文件路径：`src/frontend/`
- 前置依赖：T12、T13
- 具体动作：构建对局页面、棋盘组件、回合状态展示与基础操作区
- 完成定义：完整跑通用户主链路
- 验证方式：前端冒烟测试 + 人工走通一局最小闭环

## 12. 执行策略
### MVP 优先策略
先把“登录 → 新建一局 → 用户合法落子 → AI 合法应对 → 基础演绎展示 → 认输 / 悔棋”做成，再往主题、偏好、后台和审计扩。

### 增量交付策略
优先按 Task Bundle 派发给 Coding Agent，每次以一个 bundle 或半个 bundle 为单位回收结果，不把整份 tasks 一次性全部交出去。

### 回退策略
若规则适配层基线无法满足关键规则场景，应优先回退修订 RuleAdapter 与测试集，不直接放松 spec；若后台模型配置与未配置状态处理不清，则优先回退到后台配置流设计。
