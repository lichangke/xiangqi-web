## 1. 背景与目标
### 背景
当前常见的网页象棋产品，大多只覆盖“能下棋”这一基础能力，缺少以下组合体验：
- 严格、可信的中国象棋规则裁判能力
- 面向普通用户的人机对战闭环
- 有人格、有戏剧感的 AI 对局演绎体验
- 同时兼顾桌面端与手机端可用性
- 为后续联机、存档与 Qt 桌面端扩展预留架构空间

本项目希望以一个真正可玩的 V1 产品为起点，既验证产品方向，也验证后续可扩展架构。

### 目标
本轮目标是交付一个可上线运行的 V1 规格基线，使用户能够在桌面端和手机端使用账号登录后，进行严格规则约束下的人机中国象棋对战，并获得统一风格的 AI 棋子人格讨论、非法落子提示、悔棋、认输、终局评价等完整体验。

同时，本轮规格需明确：
- 规则正确性由硬规则层保证
- AI 价值主要体现在策略表达与戏剧演绎
- V1 暂不做人对人联机，但架构需预留扩展位
- V1 暂不做 Qt 客户端本体，但架构需避免将核心逻辑绑死在 Web UI

## 2. 用户场景 / 用户问题
### 目标用户
1. 想在网页上轻量体验中国象棋的人机对战用户
2. 对 AI 戏剧化互动有兴趣、希望“下棋过程本身也好看”的用户
3. 需要账号、偏好保存和基础对局记录的普通注册用户
4. 作为产品维护者，需要通过后台管理用户、模型配置和运行策略的管理员

### 使用场景
1. 用户在电脑端打开网页，登录账号，选择难度后开始与 AI 对战
2. 用户在手机端打开网页，快速进入棋局，通过折叠讨论区查看 AI 的角色化讨论
3. 用户尝试落出非法棋步时，系统用轻量、有风格的方式提示其不合法
4. 用户在对局中触发悔棋、认输、将军、终局等事件时，系统统一以演绎风格展示事件反馈
5. 管理员在后台创建用户、重置密码、配置模型参数、调整并发阈值与系统策略

### 当前问题
1. 现成象棋产品常见“规则有、氛围无”，缺乏有辨识度的 AI 互动体验
2. 若直接依赖大模型决定落子，难以保证合法性与规则严谨性
3. 若同时追求联机、桌面端、移动端、后台、AI 演绎，项目边界容易失控
4. 没有先定义好 V1 的用户系统、后台管理与演绎展示边界，后续 Plan 难以收口

## 3. 用户故事（按优先级）

### 用户故事 1（P1）
- 故事描述：作为普通用户，我希望在网页上与 AI 进行一局严格遵守中国象棋规则的人机对战，并且每一步都能看到风格统一、角色化的简短讨论与反馈。
- 优先级原因：这是产品的核心主链路，没有它就没有 V1 的成立基础。
- 独立价值：即使没有联机、存档和复杂后台，这条链路本身也能独立成立并体现产品特色。
- 独立验证方式：用户能够登录、开始对局、完成多步合法落子、看到 AI 正常合法应对与角色化演绎，并最终正常结束对局。

### 用户故事 2（P1）
- 故事描述：作为普通用户，我希望系统能严格阻止非法落子，并用轻量、风格化但不冒犯的方式提示我错误原因。
- 优先级原因：规则可信性是整款产品的基石，若非法落子都拦不住，AI 演绎再有趣也无法成立。
- 独立价值：单独保证规则正确性，就能显著提升产品基础质量与可信度。
- 独立验证方式：针对马腿、象眼、炮架、将帅照面、自陷将等典型场景，系统均能正确阻止并提示。

### 用户故事 3（P1）
- 故事描述：作为普通用户，我希望可以悔棋和认输，并且这些事件也能被纳入统一的演绎展示体系中。
- 优先级原因：悔棋与认输是对局体验的核心辅助能力，直接影响可玩性与完整度。
- 独立价值：即便只实现人机对战，悔棋和认输也能让主链路更完整。
- 独立验证方式：用户在规则允许条件下成功悔棋，系统正确回退完整回合并展示事件反馈；认输后正确结束对局并展示终局评价。

### 用户故事 4（P2）
- 故事描述：作为普通用户，我希望账号能保存我的基础偏好和最近对局记录，这样下次回来不需要从零开始。
- 优先级原因：这是保留普通用户登录价值的关键支撑。
- 独立价值：即使不做联机，也能使账号体系合理成立。
- 独立验证方式：用户重新登录后能够看到保存的主题偏好、基础显示偏好和最近对局记录。

### 用户故事 5（P2）
- 故事描述：作为管理员，我希望通过后台管理用户、模型配置和运行策略，以便控制系统安全性、成本和可运维性。
- 优先级原因：本项目依赖模型服务，若没有基本后台能力，运行成本、账号管理与策略调整都会很被动。
- 独立价值：后台功能可与前台主链路配套形成一个最小可运维产品。
- 独立验证方式：管理员可创建 / 禁用用户、重置密码、修改模型配置并执行连通性测试、调整并发策略并生效。

### 用户故事 6（P3）
- 故事描述：作为未来维护者，我希望当前架构为后续联机对战、棋局存档恢复和 Qt 客户端接入预留清晰边界。
- 优先级原因：这是架构约束，不是 V1 直接可见功能，但会明显影响后续演进成本。
- 独立价值：即使 V1 不立刻用到，也能降低未来返工风险。
- 独立验证方式：Plan 阶段与实现设计中能明确看到规则层、策略层、演绎层、用户 / 配置服务与 Web UI 的边界分离。

## 4. 验收场景
### 场景 1：用户开始一局标准人机对战
- Given：用户已登录且账号处于启用状态
- When：用户进入对局页并选择一个难度开始新对局
- Then：系统应创建一局新的人机对战，展示棋盘、基础操作区、讨论区入口和当前难度信息，用户可进行首次合法落子

### 场景 2：用户走出非法棋步
- Given：当前棋局处于用户回合，且用户选择了一个不合法目标点
- When：用户提交该落子
- Then：系统不得改变棋局状态，应给出轻量、风格化的不合法提示，并保持用户可继续落子

### 场景 3：AI 合法落子并输出演绎内容
- Given：用户完成一次合法落子
- When：系统进入 AI 回合
- Then：规则层先给出合法候选步集合，AI 最终落子必须属于合法集合，界面应按回合结构展示角色化讨论与最终落子说明

### 场景 4：用户执行悔棋
- Given：当前棋局允许悔棋，且本局悔棋次数未达上限，也不存在“刚悔完立即再悔”的情况
- When：用户点击悔棋并确认
- Then：系统应撤销最近一个完整回合（用户一步 + AI 一步），更新棋局、剩余悔棋次数和事件演绎展示

### 场景 5：用户认输结束对局
- Given：当前棋局尚未结束
- When：用户点击认输并完成二次确认
- Then：系统应立即结束对局、记录结果，并以与本局主题一致的风格展示终局评价

### 场景 6：用户重新登录后查看偏好与最近对局
- Given：用户此前已有历史使用记录
- When：用户重新登录系统
- Then：系统应恢复其保存的偏好设置（包括主题等基础偏好），并展示最近对局记录摘要

### 场景 7：管理员管理模型配置
- Given：管理员已登录后台
- When：管理员新增或修改模型配置并执行连通性测试
- Then：系统应保存配置、屏蔽敏感信息的非必要暴露，并返回测试结果摘要；运行期以后台持久化配置作为主配置源

### 场景 8：同一账号跨设备登录与进行中对局限制
- Given：同一账号已在一个设备上登录并存在一局进行中的对局
- When：该账号在另一设备上登录系统并尝试新开一局对局
- Then：系统可允许其登录，但应阻止创建第二局进行中的对局，并给出友好提示

## 5. 边界情况 / 异常场景
- 当模型调用超时、失败或返回不符合预期时：系统不得产出非法落子，应给出降级提示或重试策略，并保持棋局状态一致
- 当用户尝试连续悔棋时：系统应阻止该操作，并提示需先完成一次新的正常落子
- 当用户本局悔棋次数已用尽时：系统应阻止继续悔棋，并给出清晰提示
- 当同一账号存在超出系统限制的进行中对局时：系统应拒绝新对局创建并给出友好提示
- 当用户在手机端使用时：讨论区默认不应长期遮挡棋盘，非法提示不应使用重模态打断主链路
- 当后台模型配置错误时：系统应能通过连通性测试发现问题，而不是等到前台实际对局时才被动暴露
- 当规则附录细则尚未完全实现时：不得用模糊行为替代正式规则，应在实现前以专项规则附录和测试集补齐

## 6. 功能需求（FR）
- FR-001：系统必须支持基于用户名 + 密码的普通用户账号登录；V1 不要求手机号和邮箱作为必填登录或注册字段。
- FR-002：系统必须支持管理员创建用户、启用 / 禁用用户和重置密码。
- FR-003：V1 默认不得开放公网自由注册；架构需预留邀请码注册能力。
- FR-004：系统必须支持用户在桌面端与手机端访问和使用主链路。
- FR-005：系统必须允许同一账号在电脑端与手机端同时登录，但同一时刻只允许存在 1 局进行中的对局。
- FR-006：系统必须支持用户发起一局新的 AI 人机对战，并选择难度档位：新手、普通、困难、大师。
- FR-007：系统必须提供 3 种内置页面主题，并允许用户在设置中切换。
- FR-008：系统必须支持用户保存基础偏好配置，包括主题偏好、棋盘朝向 / 显示偏好、讨论区展开偏好，以及必要时的简单演绎风格偏好。
- FR-009：系统必须记录基础对局记录摘要，至少包括：开始 / 结束时间、结果、难度、是否认输结束、悔棋使用次数。
- FR-010：系统必须通过硬规则层判定用户落子合法性，并阻止任何非法落子进入棋局状态。
- FR-011：系统必须严格支持中国象棋主链路规则，包括标准走法、马腿、象眼、炮架、九宫 / 河界限制、将军判定、将帅不能照面、自陷将非法步、将死 / 无合法走法终局、认输与悔棋规则。
- FR-012：系统必须以《中国象棋竞赛规则》的现行正式版本作为规则附录的最终权威依据。
- FR-013：系统必须通过统一规则适配层向业务层提供合法走法生成、落子校验、终局判定与悔棋等规则能力。
- FR-014：AI 的最终落子必须受合法走法集合硬约束，不得输出非法步。
- FR-015：系统必须对用户非法落子给出轻量、风格化、但不冒犯的提示。
- FR-016：系统必须支持悔棋；每次悔棋应撤销最近一个完整回合，且每局最多 5 次，不允许连续悔棋。
- FR-017：系统必须支持认输，并在认输后立即结束对局。
- FR-018：系统必须为每个有效回合展示角色化、戏剧化、可公开的演绎讨论，而不是直接展示大模型原始完整思维链。
- FR-019：每个有效回合的展示必须以“第 N 回合”开头，并包含：用户落子简评、2～4 个角色发言、1 句共识 / 拍板、AI 最终落子与简短理由。
- FR-020：系统必须为非法落子、悔棋、认输、将军、终局等特殊事件提供独立展示模板。
- FR-021：系统必须保证所有棋子角色（兵、马、炮、车、士、象、帅）在整局或整体产品体验中均有出场机会，但不要求每回合全员发言。
- FR-022：系统必须保证每局固定一个演绎主题，局内不切换主题，终局评价与整局风格保持统一。
- FR-023：系统必须在后台提供模型配置管理，包括模型名称、API Key、Base URL、思考强度、启用 / 停用与连通性测试。
- FR-024：系统必须支持部署层的 env-like 初始配置引导；当系统首次无后台配置时，可从 env-like 配置初始化；进入运行期后，应以后台持久化配置作为主配置源。
- FR-025：系统应支持后台区分“对局决策模型”和“演绎模型”是否使用同一套配置。
- FR-026：系统必须提供运行策略配置，包括 AI 对局并发上限、单账号同时进行中的对局数上限、注册策略开关与系统默认值。
- FR-027：系统必须提供轻量审计与观察能力，至少记录用户创建、密码重置、模型配置修改、关键系统策略修改和模型连通性测试摘要。
- FR-028：手机端必须以棋盘为视觉中心；讨论区默认折叠，横屏时允许侧边常驻；主题切换放入设置中。
- FR-029：V1 必须采用“点击选子 + 点击目标点”的落子交互，不做拖拽落子。

## 7. 非功能需求（NFR，可选）
- NFR-001：系统不得将敏感信息（如 API Key、环境变量、密钥文件）暴露到前端或提交到仓库；项目需通过 `.gitignore` 与配置约定避免敏感信息入库。
- NFR-002：系统的规则正确性必须优先于演绎趣味性；任何情况下都不得为了戏剧效果牺牲落子合法性。
- NFR-003：手机端主链路必须可用，且讨论区、提示层与操作区不得长期压制棋盘主区域。
- NFR-004：演绎内容需控制篇幅，每回合默认控制在约 120～180 字以内，以控制阅读负担、模型成本和移动端体验。
- NFR-005：架构设计需保持规则层、策略层、演绎层、用户 / 配置服务与 Web UI 的边界清晰，以支撑后续联机、棋局恢复与 Qt 客户端扩展。
- NFR-006：后台必须具备最小可运维性，管理员应能在不改代码的情况下完成基础用户管理、模型配置与运行策略调整。
- NFR-007：V1 规则层可优先评估成熟 JS 规则库作为 baseline，首选评估 xiangqi.js，但第三方库必须通过自定义适配层接入，且其默认行为不得凌驾于官方规则依据与专项测试集之上。
- NFR-008：后台对敏感模型配置应默认采用掩码显示或非完整回显策略，避免在管理界面中暴露完整敏感值。

## 8. 关键实体 / 核心对象（必要时）
- 用户（User）：系统登录主体，具备账号身份、状态、偏好设置与基础对局记录关联
- 对局（GameSession）：一局人机象棋对战的核心会话，包含局面状态、当前回合、难度、结果、悔棋次数与事件流
- 规则适配层（RuleAdapter）：对外提供合法走法生成、落子校验、终局判定、悔棋等统一接口的规则服务层
- AI 策略与演绎配置（ModelConfig / RuntimePolicy）：描述模型连接信息、启停状态、思考强度与系统运行策略
- 回合演绎事件（NarrativeTurn / NarrativeEvent）：承载普通回合展示与特殊事件模板展示的数据结构

## 9. 范围边界
### 本轮明确包含
- 严格规则约束下的人机中国象棋对战
- 用户登录、偏好保存、基础对局记录
- 后台用户管理、模型配置、运行策略、轻量审计与观察
- AI 角色化演绎、非法落子提示、悔棋、认输、终局评价
- 桌面端与手机端主链路可用
- 为联机、棋局恢复、Qt 客户端预留架构边界

### 本轮明确不包含
- 人人对战
- 房间 / 邀请 / 匹配
- 联机实时同步
- 观战模式
- 正式的棋局保存与恢复能力
- Qt 桌面客户端本体
- 开放公网自由注册
- 好友系统、排行榜、成就系统、复杂战绩统计、复杂运营分析平台
- 读秒 / 用时制

## 10. 成功标准（SC）
- SC-001：用户能够在桌面端和手机端完成一局从开始到结束的人机对战主链路，包括合法落子、AI 合法应对、讨论展示、悔棋 / 认输与终局评价。
- SC-002：规则层能够正确阻止关键典型非法步，并通过专项测试覆盖主链路规则场景。
- SC-003：AI 展示效果具备统一风格，普通回合与特殊事件模板结构清晰，且不会明显淹没棋局本身。
- SC-004：普通用户登录后能够保存并恢复基础偏好，并查看最近对局记录摘要。
- SC-005：管理员能够通过后台完成用户管理、模型配置、并发策略调整与模型连通性测试。
- SC-006：项目实现结构能够体现规则层、策略层、演绎层与 UI / 后台的明确分层，为后续 Plan 提供足够清晰的设计基础。
